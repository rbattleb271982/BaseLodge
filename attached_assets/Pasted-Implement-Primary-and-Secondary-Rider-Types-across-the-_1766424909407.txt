Implement Primary and Secondary Rider Types across the app, aligned with existing terrain multi-select behavior, while keeping the UI mobile-first and simple.

Decisions (locked):
	•	Primary rider type is required
	•	Secondary rider types are optional, max 2
	•	Rider types are profile-level only (not trip-level)
	•	Public display never shows “Primary” or “Secondary” labels
	•	Existing rider_type field is deprecated but must be kept for backward compatibility

Authoritative Rider Type List (use exact strings only):
	•	Skier
	•	Snowboarder
	•	Telemark
	•	Cross-Country
	•	Adaptive

Database:
	•	Add primary_rider_type (String, required) to User
	•	Add secondary_rider_types (JSON, nullable) to User
	•	Keep existing rider_type column but stop writing to it
	•	Backfill existing users:
	•	primary_rider_type = rider_type
	•	secondary_rider_types = null
	•	Do not remove rider_type yet

Onboarding:
	•	Replace current single rider_type selection with:
	•	Primary Rider Type (required, single-select via radio or segmented control)
Label: “Primary rider type”
Subtext: “The way you ride most often”
	•	Secondary Rider Types (optional, multi-select via checkboxes)
Label: “Secondary rider type (optional)”
Subtext: “Only if you ride another way sometimes”
	•	Secondary list must dynamically exclude the selected Primary
	•	Enforce max of 2 secondary selections

Profile / Settings Edit:
	•	Mirror onboarding behavior exactly
	•	Inline editing only (no modal)
	•	If user changes Primary:
	•	Automatically remove that value from secondary_rider_types if present
	•	Validate before save:
	•	Primary is required
	•	Secondary count ≤ 2
	•	No overlap between primary and secondary

Backend Handling:
	•	Stop reading or writing rider_type in new logic
	•	All new logic must use primary_rider_type and secondary_rider_types
	•	Validation errors should block save with clear messaging

Display Logic (ALL locations):
	•	If only primary exists, display: “Skier”
	•	If primary + secondary exists, display: “Skier · Snowboarder”
	•	Apply this format everywhere rider type appears:
	•	Home profile card
	•	Friend profile page
	•	Friend list rows
	•	Invite landing and invite cards
	•	Any other identity rows
	•	Never show labels like “Primary” or “Secondary” publicly

Constraints:
	•	Do not add new tables
	•	Do not add frequency scoring or weighting yet
	•	Do not rename rider types
	•	Do not add trip-level rider type overrides

This implementation should fully replace the old single rider_type behavior while remaining backward compatible and safe for seeded users.
