#166 - Fix PostHog Python v7 API Compatibility

What & Why

PostHog connectivity works, but all server-side events are failing because analytics.py is using the old PostHog Python v2 method signatures while the installed package is posthog v7.12.0.

Current broken calls:

client.capture(distinct_id, event, props)

But PostHog v7 requires:

client.capture(
    event=event,
    distinct_id=str(distinct_id),
    properties=props
)

Also:

client.identify(...)

is no longer valid in v7. Use the supported v7 person-property methods instead.

Goal:

Update analytics.py to use PostHog v7-compatible calls so all server-side events reach PostHog.

REQUIRED FIXES
--------------------------------------------------

1. Fix track()

Update ph_analytics.track() so it calls:

client.capture(
    event=event,
    distinct_id=str(distinct_id),
    properties=properties or {}
)

Then call:

client.flush()

after capture.

Keep exception safety:

- Log warning on failure
- Never break the app request

Keep no-op behavior when POSTHOG_KEY is missing.

--------------------------------------------------
2. Fix identify()

PostHog v7 removed the old identify() method.

Update ph_analytics.identify() to use the correct v7-compatible method for setting person properties.

Use the safest supported approach available in the installed library.

Preferred if supported:

client.set(
    distinct_id=str(user_id),
    properties=properties or {}
)

and for set_once properties:

client.set_once(
    distinct_id=str(user_id),
    properties=set_once_props or {}
)

If the installed library uses a different v7 API, inspect it directly and use the actual supported method names.

Do not guess.

Keep exception safety.

Call flush after successful person-property updates if supported.

--------------------------------------------------
3. Preserve alias()

Keep alias() only if the installed library signature still works.

Audit the installed v7 signature before changing it.

If current alias() is valid, leave it alone.
