You are working on the BaseLodge Flask web app. Implement the following feature end-to-end, testing all flows before finishing. Use checkpoints sparingly.

Goal
Add explicit trip duration to trips so users can easily discover who is going up for a day trip, filter trips by duration, prevent duplicate overlapping trips, and support future notifications. This is a social coordination feature, not just metadata.

Data Model Changes
Update the Trip model to include a new required field called trip_duration.
Use an enum or string type consistent with the existing codebase.

Allowed values (exact):
DAY_TRIP
ONE_NIGHT
TWO_NIGHTS
THREE_PLUS_NIGHTS

Display labels:
Day Trip
1 Night
2 Nights
3+ Nights

Trip Creation and Edit Behavior

Create Trip
Require users to explicitly select a duration.
Do not infer duration from dates.
Block saving if duration is missing.

Edit Trip
If trip dates change, auto-update duration based on nights:
Same start and end date → DAY_TRIP
1 night → ONE_NIGHT
2 nights → TWO_NIGHTS
3 or more nights → THREE_PLUS_NIGHTS

Do not show a confirmation modal.
Update silently.

Active Trip Definition
Active trips are upcoming or current only.
Past trips must be ignored for overlap validation, social counts, and filters.
Treat dates as date-only. Ignore timezones entirely.

Overlapping Trip Validation

Rule
One active trip per user per resort per overlapping date range.

Scope
Overlap checks apply only to active trips.
Rule is enforced per resort.
Multiple resorts in the same weekend is allowed.

Behavior
If a user attempts to create or edit a trip that overlaps with an existing active trip at the same resort, block the action and show this alert:

“You already have a trip to {Resort Name} that overlaps these dates. Want to update that trip instead?”

Buttons:
Edit existing trip → redirect in the same tab to the edit page of the overlapping trip
Cancel → remain on the current form

Do not allow duplicate active trips.

Trip Card UI

Wherever trips appear (home, trip lists, friend cards), display duration clearly and compactly.

Examples:
Sat, Jan 13 · Day Trip
Jan 13–14 · 1 Night

Duration must be visible without clicking and should be more prominent than dates in social contexts.

Home Screen Social Signal

Add a subtle home screen card that answers “Who’s doing a day trip?”

Rules:
Weekend is defined as Friday through Sunday
Destination name must always be shown
Example copy: “3 friends are day-tripping Breckenridge this weekend”
Only show the card if the count is greater than zero
No CTA button

Filtering

Add a duration filter to the Trips list only (not friends yet).

Filter options:
Any
Day Trip
1 Night
2 Nights
3+ Nights

Filtering must:
Only include active trips
Avoid double-counting users

Notification-Ready Structure

Do not implement notifications yet.
Ensure queries can later support messages like “3 people are day-tripping Breckenridge tomorrow.”
No duplicates. No past trips.

Copy Rules

Use only the wording “Day Trip.”
Do not introduce alternate phrasing.

Testing Requirements

Before marking complete, manually test:
Creating trips with all durations
Duration is required on creation
Duration auto-updates on date edit
Overlapping trip blocking works correctly
Edit redirect works in the same tab
Home screen day-trip count accuracy
Home screen card hides when count is zero
Duration filters work correctly
No duplicate users are counted

Final Instruction

Do not mark this task complete until all scenarios are tested.
When finished, summarize what changed, what was tested, and any assumptions made.
Do not add features beyond this spec.
