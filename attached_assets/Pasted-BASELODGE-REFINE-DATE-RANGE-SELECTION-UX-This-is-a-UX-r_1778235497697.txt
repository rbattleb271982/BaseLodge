BASELODGE — REFINE DATE RANGE SELECTION UX

This is a UX refinement to the add/edit trip calendar flow.

The current interaction became too automatic.

Current behavior:
- user taps start date
- user taps end date
- range instantly auto-stages/saves
- calendar selection immediately resets

Problem:
The user never visually experiences the completed selected range state on the calendar.

This creates:
- abrupt/uncertain interaction feel
- “did it save?” confusion
- accidental range creation
- confusing single-day behavior (double-tap to create)
- lack of visual confirmation

━━━━━━━━━━━━━━━━━━━━
NEW INTERACTION MODEL
━━━━━━━━━━━━━━━━━━━━

The flow should become:

1. User taps start date
2. User taps end date
3. The FULL selected range remains visibly highlighted on the calendar
4. A lightweight inline confirmation area appears directly beneath the calendar
5. User taps “Add this range”
6. THEN:
   - the range stages into the selected list
   - the calendar resets
   - helper text appears:
     “Tap a start date to add another range.”

━━━━━━━━━━━━━━━━━━━━
1. REMOVE INSTANT AUTO-STAGING
━━━━━━━━━━━━━━━━━━━━

Do NOT automatically stage/save the range immediately after end-date tap anymore.

Instead:
- preserve the visible highlighted range
- preserve the start/end visual state
- wait for explicit inline confirmation

━━━━━━━━━━━━━━━━━━━━
2. ADD INLINE RANGE CONFIRMATION
━━━━━━━━━━━━━━━━━━━━

Directly beneath the calendar, render a lightweight inline confirmation block after a valid range is selected.

Example:

May 24 → May 26 · 2 nights

[ Add this range ]

Visual requirements:
- minimal
- editorial
- mobile-first
- calm/native feeling
- NOT a giant CTA
- NOT a heavy card
- NOT a modal

Think:
inline helper action
NOT:
form submit section

━━━━━━━━━━━━━━━━━━━━
3. SINGLE-DAY TRIP UX
━━━━━━━━━━━━━━━━━━━━

Current double-tap behavior for single-day trips is confusing.

Improve this interaction.

Preferred behavior:
- selecting the same date as start + end should clearly show:
  “May 24 → May 24 · 0 nights”
- user still explicitly confirms with:
  “Add this range”

Do NOT:
- instantly auto-stage on double tap
- create ambiguous accidental single-day trips

━━━━━━━━━━━━━━━━━━━━
4. STAGING BEHAVIOR AFTER CONFIRMATION
━━━━━━━━━━━━━━━━━━━━

Only AFTER user taps “Add this range”:
- add range to staged list
- clear active calendar selection
- reset calendar into “ready for another range” state

Then show helper text:
“Tap a start date to add another range.”

Preserve:
- remove buttons on staged ranges
- overlap validation
- duplicate validation
- grouped/batch trip logic
- dynamic CTA (“Save trip”, “Save 2 dates”, etc.)

━━━━━━━━━━━━━━━━━━━━
5. MOBILE-FIRST UX
━━━━━━━━━━━━━━━━━━━━

This should feel:
- visually understandable
- calm
- deliberate
- app-like
- confidence-building

The user should always clearly understand:
- what they selected
- whether it has been staged yet
- what action happens next

━━━━━━━━━━━━━━━━━━━━
GUARDRAILS
━━━━━━━━━━━━━━━━━━━━

Do NOT:
- redesign the page
- change typography system
- change colors
- change bottom nav
- introduce Tailwind
- remove batch trip logic
- break group flow
- remove existing validations

This is specifically an interaction refinement.

Use checkpoints sparingly.

Do not finish until you have tested:
- normal multi-day range
- single-day range
- adding multiple ranges
- removing staged ranges
- overlap validation
- duplicate validation
- group flow still working
- CTA counts still updating correctly
