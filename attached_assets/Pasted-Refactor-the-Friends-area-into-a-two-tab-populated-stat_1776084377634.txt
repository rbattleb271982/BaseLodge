Refactor the Friends area into a two-tab populated-state experience, while preserving the current empty state for users with no friends.

This is a larger feature pass, so please implement it carefully end-to-end, keep the existing editorial visual system, and make the result mobile-first. Use checkpoints sparingly. Do not overuse them.

==================================================
GOAL
==================================================

We need the Friends section to support two very different states:

1. EMPTY STATE
   - For users with zero friends/connections
   - Keep the current invite / QR code experience
   - But update copy so it is inclusive and not ski-only

2. POPULATED STATE
   - For users with 1+ friends/connections
   - Replace the current single-screen Friends experience with two top tabs:
     - Friends
     - Friends’ Trips

This should feel like one cohesive system:
- Friends = people / directory / relationship layer
- Friends’ Trips = where friends are going / context layer
- Ideas remains the place for overlap/recommendation logic
- Do not make Friends’ Trips into a noisy activity feed

==================================================
HIGH-LEVEL PRODUCT RULES
==================================================

1. Do NOT say “ski crew” anywhere in user-facing copy
2. Do NOT use male-coded group language like “crew” or “squad”
3. Do NOT make this feel like a social media feed
4. Do NOT duplicate Ideas tab functionality
5. Keep the visual style calm, premium, editorial, and clean
6. Mobile-first first, desktop second
7. Preserve the current tone of the app and existing design language
8. Do not add cards, pills, badges, avatars everywhere, or noisy UI
9. Search belongs in the Friends directory tab, not in Friends’ Trips
10. Invite remains important, but should not dominate the populated state

B) If user has 1 or more friends/connections:
- Show the new two-tab Friends experience

  - Friends if that is easiest with current implementation
  - but structure code so default can be changed later if needed

==================================================
POPULATED STATE: TOP-LEVEL STRUCTURE
==================================================

For users with 1+ friends, redesign the Friends page with this structure:

Header area:
- Eyebrow or small label: FRIENDS
- Large title: Friends
- Invite action on the top right as simple text link/button: Invite

Under title:
- Two top tabs:
  - Friends
  - Friends’ Trips

These tabs should feel editorial and lightweight, not like chunky segmented controls.
Use the current visual language of hairlines, text emphasis, and calm spacing.

==================================================
TAB 1: FRIENDS
==================================================

Purpose:
- Directory / relationship layer
- Manage and view people
- Should feel alive enough to be useful, but not like a feed

This tab should include:

1. Search bar directly under the tab row
   - Placeholder should be neutral and inclusive
   - Use:
     - “Search friends”
   NOT:
     - “Search your crew”
   - Search should feel calm and editorial, not like a giant app-store style search field
   - Search should filter across at least:
     - friend name
     - pass
     - level
   - When actively filtering:
     - alphabetical letter headers disappear
     - filtered rows show clean result list
   - If no results:
     - show a quiet inline message like:
       - No friends match “X”.
     - Keep tone calm, no loud empty-state treatment

2. Alphabetical grouped friend list
   - Group by first letter
   - Show letter header
   - Then friend rows beneath

3. Friend row content
   Keep rows simple and useful. Recommended structure:
   - Left:
     - subtle circular initials marker OR existing light identity marker if already present
     - friend name
     - optional secondary line with concise profile info like level + pass
   - Right:
     - quiet trip summary, for example:
       - 2 trips
       - 1 going
       - or “No trips”
   Keep this compact and editorial, not dashboard-like

4. Invite action
   - Keep Invite in header top-right
   - Do not show giant QR block in populated Friends tab
   - Tapping Invite should open the invite experience / QR experience / existing invite flow
   - Reuse current QR/invite UI in a modal, drawer, overlay, subview, or dedicated invite screen if needed
   - Main populated Friends tab itself should not be dominated by QR code

5. Tapping a friend row
   - If a friend detail page already exists, route there
   - If not, keep the row ready for future drill-in without breaking the current build

Important constraints for Friends tab:
- Do not turn this into a feed
- Do not add recent activity as the primary content
- This tab is mainly for people lookup, relationship context, and lightweight scanning

==================================================
TAB 2: FRIENDS’ TRIPS
==================================================

Purpose:
- Calm, destination-first view of where friends are going
- Not a feed
- Not recommendation logic
- Not overlap-first
- Ideas tab still owns overlap/recommendation logic

Core content model:
- Group by month
- Within each month, show destination-first rows
- This should answer:
  - “Where are my friends going?”
  - not
  - “What should I do?”

Structure:
- Month grouping headers (e.g. December, January, February)
- Under each month, trip rows

Each trip row should use this exact hierarchy:

Line 1:
- Destination only
- Example:
  - Vail

Line 2:
- Left side:
  - [Friend name] · [status] · [new if applicable]
- Right side:
  - [Date]
- Example:
  - Sam · going · new        Dec 5–7

Important:
- The date is NOT on the same line as the mountain
- The date belongs on the same row as the friend/status metadata
- Use flexbox or equivalent so metadata is left-aligned and date is right-aligned
- Keep the mountain line visually pure and uninterrupted

Statuses:
- Use:
  - planning
  - going
- Do NOT use:
  - booked

“new” indicator:
- Support a subtle lowercase “new” indicator for recently created trips
- Show “new” only for trips created within the last 7 days
- Do NOT show “new” for edits or status changes
- Keep “new” lowercase
- Keep “new” visually subtle and editorial
- Italic serif is okay
- Do NOT use pills, badges, dots, or loud notification styling
- Place it inline in the metadata line:
  - Sam · going · new
- Do not move “new” to the far left
- Do not float it by itself on the far right

Date styling:
- Right-aligned
- Secondary
- Slightly muted color vs primary mountain text
- Do not make date heavier than destination
- Date should support scanability, not compete with destination

Friends’ Trips tab constraints:
- Do not show pass type here
- Do not show skill level here
- Do not emphasize overlap here
- Do not turn rows into a noisy activity feed
- Do not add avatars or icons to every row
- Keep it clean and scannable
