PUT THIS IN PLAN FIRST

Fix onboarding rider-type selection styling consistency, eliminate accidental onboarding scrolling/overflow issues, restore the full ski pass list on the onboarding ski pass step, and correctly represent rental equipment users throughout the app so the onboarding flow feels fully native and logically consistent on mobile.

Context:
This is the onboarding flow where users:
1. Select rider types
2. Select ski passes
3. Configure equipment preferences

Current issues:

1. Selected-state inconsistency
- When 1–2 rider types are selected, selected options correctly appear with:
  - bordeaux filled background
  - white text
  - active selected styling
- When a 3rd option is selected, it does NOT retain the same highlighted appearance.
- This creates uncertainty and makes the user think the 3rd selection may not have saved.

2. Onboarding scroll/overflow behavior
- The onboarding screens can scroll horizontally.
- The onboarding screens can vertically bounce/scroll even when all content fits on screen.
- This makes onboarding feel like a web page instead of a native app experience.

3. Ski pass onboarding regression
- The ski pass onboarding screen currently only shows:
  - No pass
  - Not sure yet
  - Epic
  - Ikon
  - Other pass
- The rest of the ski passes are missing.
- This appears to be using a shortened onboarding-only list instead of the canonical pass source.

4. Rental equipment status bug
- If a user selects that they rent skis/equipment during onboarding, the Home tab later shows:
  “No equipment”
- This is incorrect.
- Renting equipment is still a valid equipment state and should not be treated as having no equipment setup.

==================================================
RIDER TYPE SELECTED STATE FIXES
==================================================

Required behavior:
- Every selected rider type must use the exact same active visual state regardless of how many are selected.
- Selected state should always include:
  - bordeaux filled background
  - white text
  - same border treatment
  - same font weight
- Do not visually dim, mute, soften, or downgrade selections after 2.
- Keep the helper copy:
  “Choose what fits — most people pick two.”
- Do not enforce a hard max selection unless backend validation already requires it.
- If any frontend-only “max 2” logic exists, remove it.

==================================================
ONBOARDING SCROLL / OVERFLOW FIXES
==================================================

Required behavior:
- Disable horizontal scrolling across all onboarding screens.
- Prevent any element from exceeding viewport width.
- Audit for:
  - width: 100vw
  - negative margins
  - oversized flex/grid widths
  - absolute-positioned elements causing overflow
  - progress bar overflow
  - fixed-width containers
- Replace width: 100vw with width: 100% where appropriate.
- Ensure onboarding wrappers use:
  box-sizing: border-box
- Add overflow-x: hidden to onboarding page wrappers/body as a safeguard.

==================================================
VERTICAL LAYOUT BEHAVIOR
==================================================

Required behavior:
- Onboarding should feel fixed and app-native.
- If content fits within the viewport, the page should not vertically scroll or bounce.
- Use min-height: 100dvh or height: 100dvh appropriately.
- Respect iOS safe-area insets.
- Keep the Continue button visually anchored near the bottom of the screen without creating overflow.
- Do not introduce clipping on smaller devices.
