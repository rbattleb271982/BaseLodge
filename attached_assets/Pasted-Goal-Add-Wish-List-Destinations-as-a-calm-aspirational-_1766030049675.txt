Goal

Add Wish List Destinations as a calm, aspirational profile feature (editable from More, visible on profiles), including Wish List overlap between friends, and add Equipment Purchase Year as a data-only enhancement. Maintain an Apple/Airbnb-style UI: quiet, intentional, non-socially noisy.

⸻

1) Wish List Destinations

Purpose

Capture aspirational resorts users want to ski someday.
This is not trip planning, not commitment, and not a social feed.

⸻

Data Model

Add a Wish List Destinations field:
	•	List of up to 3 resorts
	•	Stored as resort IDs (preferred) or names that match existing resort data
	•	Optional
	•	Enforce max of 3 strictly

⸻

Editing Location (Important)

Wish List Destinations are edited only from the More page.

More page row:
Wish List Destinations
Secondary text (if filled):
“3 resorts”

Tap → Wish List selection screen.

⸻

Wish List Selection Screen
	•	Search or browse existing resorts
	•	Add / remove resorts
	•	Hard cap at 3 (disable add once limit reached)
	•	No ordering or prioritization
	•	No social context

⸻

Profile Display (Self + Friends)

Wish List Destinations should display on:
	•	The user’s own profile
	•	Friend profile pages

Display rules:
	•	Flat list (no cards)
	•	Calm, secondary typography
	•	Section title: Wish List
	•	Example:
• Niseko
• Zermatt
• Jackson Hole

Rules:
	•	Hide section entirely if empty
	•	No counts, badges, icons, or emojis
	•	View-only (no actions)

⸻

2) Wish List Overlap (Friend Profiles Only)

Purpose

Surface light, aspirational overlap between friends without pressure or calls to action.

⸻

Overlap Logic

When viewing a Friend Profile:
	•	Compare the current user’s Wish List with the friend’s Wish List
	•	An overlap exists if any resort appears in both lists
	•	Simple intersection only (no weighting, ranking, or counts)

⸻

Display Rules

If overlap exists, show a section above the Wish List section:

Section title (exact copy):
Wish List Overlap

Display format (flat list):
• Niseko
• Zermatt

Rules:
	•	Show only overlapping resorts
	•	Flat list, secondary typography
	•	Hide section entirely if no overlap
	•	No icons, emojis, counts, or CTAs

This is ambient insight, not an action prompt.

⸻

3) Equipment Purchase Year (Data-first, UI-minimal)

Purpose

Capture equipment lifecycle data without encouraging gear flex or adding clutter.

⸻

Data Model

For Skis / Board only (not boots), add:
	•	purchase_year (integer, e.g. 2010–current year)

Rules:
	•	Optional
	•	One value per primary riding item (Skis OR Board, depending on rider type)
	•	Minimal validation (reasonable year range only)

⸻

Editing Location

Settings → Equipment.

⸻

Display Rules

Collapsed equipment view:
	•	No change
	•	Do NOT show purchase year

Expanded equipment view (user’s own profile only):
If purchase year exists, show:

Skis: Volkl Mantra 177cm
Bought: 2021

Rules:
	•	Use label “Bought”
	•	Muted, secondary text
	•	One line
	•	Do NOT display purchase year on friend profiles, lists, or trip cards

⸻

Global Guardrails
	•	No comments, feeds, likes, or notifications
	•	No automatic trip suggestions from wish list
	•	No visual emphasis that competes with trips
	•	All new information remains secondary
	•	Profiles must remain scannable in ~3 seconds

⸻

Acceptance Criteria
	•	Users can edit Wish List Destinations from More
	•	Wish List displays cleanly on self and friend profiles
	•	Wish List Overlap appears only when applicable and feels delightful, not demanding
	•	Equipment purchase year is captured without clutter
	•	Overall UI remains calm, premium, and intentional
