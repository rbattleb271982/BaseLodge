Improve both the Create Trip and Edit Trip flows by introducing a Country-first location hierarchy, preventing invalid date selection, automatically calculating trip duration from dates, and simplifying the form by removing manual duration selection. Also add resort search for long lists.

This is a UX and data-correctness update. Do not change database schema.

⸻

SECTION 1: Defaults and data prerequisites (do first)
	1.	User country default logic
Before updating the UI, ensure there is a reliable way to determine the user’s default country.

Rules:
	•	If the user has a country set on their profile, use that
	•	Otherwise default to “United States”
	•	This default should be available to both Create Trip and Edit Trip flows

Do not introduce new fields. Reuse existing user/profile data.

⸻

SECTION 2: Location hierarchy (Country → State → Resort)
	2.	Add Country selector
Add a Country dropdown at the top of both Create Trip and Edit Trip pages, above State.

Rules:
	•	Country defaults using the logic defined in Section 1
	•	Country options come from the existing supported countries in the app
	•	Do not hard-code U.S.-only assumptions beyond the default

	3.	State selector behavior

	•	State dropdown must be disabled until a Country is selected
	•	State options must be filtered by selected Country
	•	When Country changes, reset State and Resort selections

	4.	Resort selector behavior

	•	Resort dropdown must be disabled until a State is selected
	•	Resort options must be filtered by selected State (and Country implicitly)
	•	When State changes, reset Resort selection

	5.	Backward compatibility

	•	Existing trips without a Country value should continue to render correctly
	•	Country is required only during Create/Edit going forward
	•	Trip list and trip detail views should not be modified unless already wired

⸻

SECTION 3: Resort search (after hierarchy exists)
	6.	Add resort search
Add search capability to the Resort selector in both Create Trip and Edit Trip flows.

Rules:
	•	Search filters resorts by name as the user types
	•	Search only applies within the selected State (and Country)
	•	Do not allow searching across all locations at once
	•	This is a UI enhancement only; do not change how resorts are stored

⸻

SECTION 4: Date validation (before duration logic)
	7.	Prevent selecting past dates
On both Create Trip and Edit Trip pages, enforce valid date selection.

Rules:
	•	Earliest selectable Start date is today
	•	Earliest selectable End date is today
	•	End date must not be earlier than Start date

Implementation:
	•	Enforce via date picker minimum values
	•	Validate again on form submit
	•	Block save and show inline error if invalid

Trips must not be creatable or editable with past dates.

⸻

SECTION 5: Automatic trip duration (after dates are valid)
	8.	Remove manual Trip Duration selection
Remove the Trip Duration segmented control entirely from both Create Trip and Edit Trip pages:

	•	Day Trip
	•	1 Night
	•	2 Nights
	•	3+ Nights

Trip duration should no longer be user-selectable anywhere.
	9.	Duration calculation logic
Automatically derive trip duration from Start date and End date.

Logic:
	•	If Start date and End date are the same calendar day:
	•	Duration = “Day Trip”
	•	Otherwise:
	•	Nights = difference in days between End date and Start date
	•	1 night → “1 Night”
	•	2 nights → “2 Nights”
	•	3 or more nights → “3+ Nights”

Recalculate dynamically whenever either date changes.
	10.	Usage of calculated duration

	•	Use the calculated value anywhere duration is displayed
	•	If duration is stored today, continue storing it but derive it from dates on save/update
	•	Do not require user confirmation

⸻

	11.	Mirror behavior in Edit Trip

	•	Automatic duration calculation

Create Trip and Edit Trip must behave identically.
