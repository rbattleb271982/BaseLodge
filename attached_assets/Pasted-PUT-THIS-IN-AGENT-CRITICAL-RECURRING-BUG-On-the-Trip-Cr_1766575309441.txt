CRITICAL RECURRING BUG:
On the Trip Create/Edit screen, users cannot reliably select State, Resort, or view the Calendar. Fields appear populated but remain functionally locked. This issue keeps resurfacing because form interactivity is incorrectly dependent on onChange events instead of actual data state.

GOAL:
Make State, Resort, and Calendar ALWAYS interactive when valid data exists — regardless of whether values were set by the user or loaded from an existing trip. This fix must permanently eliminate the “locked form” behavior.

ROOT CAUSE (MUST BE FIXED):
- State and Resort dropdowns are enabled ONLY via onChange handlers
- Existing trip values are rendered but NOT injected into JS state
- Calendar initialization is skipped when dates already exist
- Disabled attributes are applied based on UI assumptions, not data truth

PERMANENT FIX — REQUIRED CHANGES:

1. SINGLE SOURCE OF TRUTH FOR FORM STATE
Create a single JS object on page load:

formState = {
  country: trip.country || null,
  state: trip.state || null,
  resort: trip.resort_id || null,
  startDate: trip.start_date || null,
  endDate: trip.end_date || null
}

All enable/disable logic MUST reference formState — not DOM events.

2. FORCE FORM REHYDRATION ON LOAD
On DOMContentLoaded (or equivalent):

- Set country select value from formState.country
- Populate state options immediately if formState.country exists
- Enable state select if formState.country exists
- Set state select value from formState.state

- Populate resort options immediately if formState.state exists
- Enable resort select if formState.state exists
- Set resort select value from formState.resort

DO NOT wait for user interaction to populate or enable fields.

3. REMOVE ALL SERVER-SIDE DISABLED ATTRIBUTES
- Remove disabled attributes from State and Resort HTML
- Disabling must be controlled ONLY via JS using formState
- HTML should render all selects as enabled by default

4. CALENDAR MUST ALWAYS INITIALIZE
Calendar logic MUST run unconditionally.

- Always render calendar UI (days + arrows)
- If startDate/endDate exist, pre-select them
- If not, render empty selectable calendar

DO NOT short-circuit calendar rendering based on existing values.

Example (conceptual):
initializeCalendar({
  startDate: formState.startDate,
  endDate: formState.endDate,
  editable: true
})

5. ON CHANGE HANDLERS UPDATE STATE — NOT GATEKEEP UI
onCountryChange:
- Update formState.country
- Reset formState.state and formState.resort
- Repopulate state options
- Enable state select

onStateChange:
- Update formState.state
- Reset formState.resort
- Repopulate resort options
- Enable resort select

onResortChange:
- Update formState.resort

6. VALIDATION (ALL MUST PASS — NON-NEGOTIABLE):
- User can always open State dropdown when Country exists
- User can always open Resort dropdown when State exists
- User can change Country multiple times without refresh
- State + Resort always repopulate correctly
- Calendar always renders and is editable
- Works on Create Trip and Edit Trip
- Works on mobile Safari

DO NOT:
- Do NOT rely on visual values as logic
- Do NOT gate logic behind onChange-only flows
- Do NOT reintroduce disabled HTML attributes
- Do NOT add schema changes
- Do NOT create separate create/edit templates

DELIVERABLE:
Trip Create/Edit form is fully re-editable at all times. No locked fields. No missing calendar. This bug cannot recur.
