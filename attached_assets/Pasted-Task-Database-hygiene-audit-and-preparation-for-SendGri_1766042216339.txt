Task: Database hygiene audit and preparation for SendGrid (email now, push later)

Goal: Make sure the database and backend logic are clean, explicit, and safe to support personalized, event-driven emails using SendGrid, while remaining channel-agnostic so push notifications can be added later. This task is about data readiness only — not sending emails.

Please do the following steps in order.

Step 1 — Verify existing schema
Inspect the actual database schema (do not assume) and list all relevant tables and columns, including:
	•	table name
	•	column name
	•	data type
	•	nullable or not
	•	default value
	•	indexes or constraints

Focus specifically on:
	•	users
	•	connections or friends
	•	trips
	•	feed or activity
	•	any existing notification, email, or event-related tables

Step 2 — Check required capabilities
Compare the existing schema against the requirements below and explicitly mark each item as:
exists
partially exists
missing

A) User identity and lifecycle
	•	canonical user_id
	•	email (unique and normalized)
	•	first_name
	•	created_at
	•	last_active_at
	•	lifecycle_stage
	•	onboarding_completed_at
	•	profile_completed_at
	•	first_connection_at
	•	first_trip_created_at
	•	is_seeded or test-user flag

B) Notification preferences (channel-agnostic)
	•	email_opt_in
	•	email_transactional_enabled
	•	email_social_enabled
	•	email_digest_enabled
	•	timezone

C) Connections (for personalization)
	•	bidirectional connection records
	•	created_at timestamp on connections
	•	connection_count (stored or safely derivable)
	•	ability to identify seeded/test connections

D) Event system (single source of truth)
Confirm or add support for a dedicated events table with:
	•	event_id
	•	event_name (e.g. account_created, connection_created, trip_created)
	•	user_id
	•	payload (JSON or equivalent)
	•	created_at
	•	environment (dev or prod)

These events should be usable later for both email and push notifications.

E) Email suppression and audit
Confirm or add an email_log table that can track:
	•	user_id
	•	email_type (e.g. welcome_v1, connection_created_v1)
	•	source_event_id
	•	last_sent_at
	•	send_count
	•	environment (dev or prod)

This table should support deduplication and suppression across all email types.

Step 3 — Gap analysis and proposal
For each missing or insufficient item:
	•	propose the minimal schema change required
	•	state whether the change is additive only or requires backfill
	•	do not rename or remove existing columns

Step 4 — Implementation constraints
Implement only:
	•	additive schema changes
	•	new tables if required
	•	safe default values

Do not:
	•	drop or rename columns
	•	change existing behavior without confirmation
	•	send any emails
	•	add SendGrid-specific IDs to the schema

Step 5 — Verification
After implementation, provide:
	•	an updated schema summary
	•	example queries or ORM access showing:
lifecycle_stage can be read safely
connection_count is available for personalization
email suppression logic can be evaluated
seeded/test users are excluded from sends

Deliverable:
A short written audit summary plus implemented, non-destructive schema upgrades that make the system ready for SendGrid now and push notifications later.
