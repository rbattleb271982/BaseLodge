Goal
Refactor the Friends screen filters to use a single Filter bar with a bottom sheet on mobile. Remove inline filters entirely. The Rider Type filter must include all rider types supported by the app, with “All” as the default.

Scope and requirements
	1.	Remove inline filters
Remove the existing inline filter card from the Friends screen so that the friends list appears immediately under the Friends header. Filters should no longer take up vertical space on the main screen.
	2.	Add a filter bar at the top of the screen
Add a compact bar below the Friends header with:

	•	Left: “Filter”
	•	Middle (only when active): “• X active”
	•	Right: “Clear”

Behavior:
	•	Tapping “Filter” opens a bottom sheet
	•	Tapping “Clear” resets all filters to default (All selected) and refreshes the list

	3.	Bottom sheet filter UI
Implement a slide-up bottom sheet containing all filters.

Rider Type
Include ALL rider types supported by the app. At minimum, this must include:
	•	All
	•	Skier
	•	Snowboarder
	•	Both
	•	Adaptive
	•	Telemark
	•	Other (if present in the data model)

Do not hard-code logic that assumes only skier/snowboarder. The filter must work for any rider_type value stored on the User model.

Skill Level
Options:
	•	All
	•	Beginner
	•	Intermediate
	•	Advanced
	•	Expert

Pass
Options:
	•	All
	•	Epic
	•	Ikon
	•	Other

Footer:
	•	Clear all (resets to default)
	•	Apply filters (applies and closes)

UI behavior:
	•	Bottom sheet height approximately 85–90% of screen
	•	Scrollable content
	•	Sticky footer buttons
	•	Dismissible by swipe down or backdrop tap

	4.	Default state and checkbox rules (apply to all filter groups)

	•	All is selected by default for Rider Type, Skill Level, and Pass
	•	Selecting All selects every individual option and applies no filtering
	•	Unchecking any individual option automatically unchecks All
	•	If all individual options are checked, automatically re-check All
	•	If the user unchecks all individual options, automatically revert to All
	•	Never allow an empty filter state

	5.	Rider type filtering logic (critical)
Filtering must be inclusive and support multi-category overlap.

Rules:
	•	If All is selected, do not apply a rider type filter
	•	Otherwise, include users whose rider_type is in the selected set
	•	Additionally, users with rider_type = “both” must be included when Skier or Snowboarder is selected
	•	Do not exclude Adaptive, Telemark, or other rider types unless the user explicitly unchecks them

Example logic:
	•	Selected: Skier → include skier and both
	•	Selected: Snowboarder → include snowboarder and both
	•	Selected: Skier + Telemark → include skier, telemark, and both
	•	Selected: Adaptive only → include adaptive only

	6.	Backend filtering logic

	•	Apply filters only when selected values are a strict subset of all possible values
	•	Use inclusion logic (IN queries), not equality checks
	•	If the selected rider types include all known rider types, skip rider filtering entirely
	•	Do not assume rider types are limited to a fixed enum unless that already exists in the codebase

	7.	Active filter count

	•	Count how many filter groups are actively filtering (not in the All state)
	•	Display this as “Filter • X active”
	•	Example: Rider Type filtered + Skill Level filtered = “Filter • 2 active”

	8.	Non-goals
Do not redesign friend cards
Do not add sorting
Do not add filter chips
Do not change database schema
	9.	Testing requirements
Verify default load shows all friends
Verify filtering works for all rider types (including Adaptive and Telemark)
Verify users with rider_type = “both” appear correctly
Verify Clear and Clear all reset everything
Verify no empty results due to invalid filter state
Verify mobile viewport behavior

Agent instructions
Use minimal checkpoints
Test before marking complete
Do not refactor unrelated code
