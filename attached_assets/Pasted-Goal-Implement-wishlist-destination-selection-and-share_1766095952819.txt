Goal
Implement wishlist destination selection and shared-interest surfacing, including the Home screen “Shared Interest” card, using existing canonical resorts.
This step does NOT add notifications or social actions — only ambient discovery.
1. Scope (Strict)
In scope
Wishlist destinations stored as canonical resort_id
Wishlist selection UX (search by destination name only)
Shared-interest aggregation
Home screen Shared Interest card (collapsed + expandable)
Out of scope
Push notifications
Friend-specific matching
Trip overlap logic
Any schema refactor unless required
2. Data Model (Minimal)
If a wishlist table already exists
Use it as-is.
If not, create one simple table only:
class UserWishlist(db.Model):
    id = db.Column(db.Integer, primary_key=True)
    user_id = db.Column(db.Integer, db.ForeignKey("user.id"), nullable=False)
    resort_id = db.Column(db.Integer, db.ForeignKey("resort.id"), nullable=False)
    created_at = db.Column(db.DateTime, default=datetime.utcnow)

    __table_args__ = (
        db.UniqueConstraint("user_id", "resort_id", name="unique_user_resort_wishlist"),
    )
No other fields.
No denormalization.
3. Wishlist Selection UX
Requirements
User can add a wishlist destination by typing the destination name
Search should match:
resort.name
Display results as:
Chamonix · France
Verbier · Switzerland
User does not need to select country or region
On tap → save (user_id, resort_id) to wishlist
Prevent duplicates silently
4. Shared-Interest Aggregation Logic
Definition
A shared interest exists when:
The current user has a resort on their wishlist
AND at least one other user also has that same resort_id on their wishlist
Query logic (conceptual)
For a given user:
Get all resort_ids on their wishlist
For each, count:
COUNT(DISTINCT user_id)
Keep only resorts where count ≥ 2
5. Destination Ranking Rules (Must Follow)
When multiple shared-interest destinations exist:
Primary: highest overlap count
Secondary (tie-breaker): most recently added by the current user
Always select one top destination for the collapsed Home card.
6. Home Screen – Shared Interest Card
When to show
Show only if:
User has ≥ 1 shared-interest destination
Otherwise:
Do not show the card
No empty state
Collapsed State (Default)
Card title (small, muted):
Shared Interest
Primary copy:
You and 12 others are interested in Chamonix
Secondary hint (only if >1 shared destination):
+ 2 more destinations
Entire card is tappable
No icons
No CTA buttons
Expanded State (On Tap)
Reveal a simple list:
Shared Interests
• Chamonix · 12 others
• Verbier · 5 others
• Niseko · 3 others
Rules:
Sorted by overlap count (desc)
Each row tappable → destination detail / wishlist view
Cap list at ~5 items (no pagination)
7. Copy & Tone Rules (Important)
Neutral, calm, informational
No emojis
No urgency language
No “Trending”, “Popular”, or “Hot”
Approved phrasing:
“You and X others are interested in {Destination}”
“X others also have {Destination} on their wishlist”
8. Implementation Notes
Recompute shared-interest data:
On page load (Home)
No background jobs required
Cache not required for v1
Logic should gracefully update when:
User adds/removes wishlist items
9. Validation Checklist (Must Confirm)
Wishlist selection works by destination name only
No duplicate wishlist rows per user
Shared-interest card:
Appears only when applicable
Shows exactly one destination collapsed
Expands correctly when multiple matches exist
Removing a wishlist destination removes it from shared interest
No performance regressions
No UI clutter introduced
10. Explicit Non-Goals
❌ No notifications
❌ No friend names
❌ No alerts or badges
❌ No ranking explanation shown to users
