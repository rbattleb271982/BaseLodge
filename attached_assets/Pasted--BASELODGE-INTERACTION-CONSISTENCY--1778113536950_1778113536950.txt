║ BASELODGE INTERACTION CONSISTENCY + CARD SYSTEM REFACTOR ║
╚════════════════════════════════════════════════════════════╝

We need to standardize interactive surfaces across the BaseLodge app and remove ambiguous inline navigation arrows.

GOAL

The app currently uses small inline arrows (`→`) beside dismiss (`×`) actions inside tappable surfaces. This is confusing because the arrow is trying to communicate “this is tappable,” while the entire row/card is already interactive.

We are moving to a cleaner Apple/editorial interaction model:

1. If an entire surface is tappable:
   - the surface itself should visually communicate interactivity
   - remove standalone inline arrows
   - keep only secondary utility actions like dismiss (`×`) if needed

2. Chevron-style navigation rows (`›`) should ONLY exist in traditional list/settings navigation patterns.

We also need to strengthen visual affordances for clickable profile metadata and stat surfaces.

────────────────────────────────────
PART 1 — REMOVE REDUNDANT INLINE →
────────────────────────────────────

Audit all logged-in templates/components for:
- inline `→`
- SVG arrow icons
- row-end navigation arrows
- any arrow whose purpose is only “this is tappable”

Especially audit:
- Home feed/opportunities
- Invite modules
- Friends activity cards
- Trip opportunity modules
- Any remaining ideas/opportunity surfaces
- Cards with dismiss (`×`) actions

For tappable cards/surfaces:
- REMOVE the inline arrow entirely
- Preserve existing tap/click behavior
- Entire card/surface should remain tappable

DO NOT remove:
- traditional list-row chevrons in settings/profile navigation
- disclosure arrows for accordions
- back navigation arrows

────────────────────────────────────
PART 2 — CREATE A CONSISTENT CARD SYSTEM
────────────────────────────────────

For tappable feed/opportunity/invite surfaces:

Convert them into visually distinct cards.

Card styling direction:
- editorial/premium
- subtle tonal separation
- NOT dashboard/SaaS styling

Use:
- slightly different surface color from page background
- subtle border
- soft radius
- soft/no shadow
- proper vertical spacing

Design targets:
- feels like layered paper/surfaces
- obvious tap target
- clean mobile-first hierarchy

Suggested style direction:
- background: slightly lighter than page background
- radius: ~18–22px
- subtle border using low-opacity maroon/charcoal
- generous internal padding
- spacing between cards increased slightly

DO NOT:
- introduce harsh gray cards
- use heavy shadows
- create “material dashboard” styling

────────────────────────────────────
PART 3 — SECONDARY ACTION CLEANUP
────────────────────────────────────

Dismiss (`×`) actions should become visually secondary.

Update dismiss actions:
- smaller visual weight
- slightly reduced opacity
- aligned upper-right where appropriate
- preserve accessibility/tap target

The card content should remain the primary focus.

────────────────────────────────────
PART 4 — CLICKABLE TEXT + STAT AFFORDANCES
────────────────────────────────────

We need stronger visual affordances for clickable profile metadata.

On the profile/home identity card:

The following values are clickable and should visually look clickable:
- Skier / Snowboarder
- Skill level
- Pass type
- Skis
- Boots

Update these to:
- maroon text
- subtle underline
- preserve existing tap behavior
- maintain premium/editorial styling

DO NOT style them like aggressive web hyperlinks.
They should feel refined and intentional.

The stats row should also visually communicate interactivity.

Current stats:
- Upcoming Trips
- Mountains Visited
- Wishlist Mountains

Requirements:
- Entire stat block should be tappable
- Add subtle visual affordance that the stats are interactive
- Maintain clean editorial aesthetic
- Avoid adding arrows/icons
- Preserve current centered layout and hierarchy

Potential direction:
- subtle opacity/tone change on press
- clearer hover/tap state
- slightly darker labels or underlined labels if needed
- maintain premium Apple-like feel

────────────────────────────────────
PART 5 — INTERACTION RULES
────────────────────────────────────

Establish these interaction conventions consistently:

SYSTEM A — TAPPABLE CARDS
Used for:
- feed modules
- opportunities
- invites
- passive activity cards

Behavior:
- entire surface tappable
- NO inline navigation arrow
- optional dismiss/utility action allowed

SYSTEM B — NAVIGATION ROWS
Used for:
- settings
- profile sections
- account rows
- equipment lists
- drill-in rows

Behavior:
- keep chevron/disclosure affordance
- full row tappable
- no competing dismiss actions

SYSTEM C — CLICKABLE METADATA
Used for:
- profile identity traits
- pass types
- equipment
- stat surfaces

Behavior:
- maroon text
- subtle underline or refined affordance
- preserve premium/editorial styling
- no loud web-link aesthetics

────────────────────────────────────
PART 6 — QA
────────────────────────────────────

Verify:
- no tap behavior regressed
- dismiss actions still work
- spacing consistent across cards
- no arrows accidentally removed from true nav rows
- no layout overflow on iPhone widths
- no broken alignment in bottom-nav safe area layouts
- all interactions remain mobile-first
- clickable metadata clearly reads as interactive
- stats row interactions feel intentional and discoverable

Use checkpoints sparingly. Only use them if absolutely necessary.
