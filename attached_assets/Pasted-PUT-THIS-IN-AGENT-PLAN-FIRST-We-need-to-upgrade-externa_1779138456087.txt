==================================================
GOAL
==================================================

When a user taps:

“Invite friends to this trip”

and chooses:

“Share via text” or “Copy invite link”

the shared link should point to that specific trip, not just the inviter’s generic friend invite link.

Current issue:

External share currently uses:

/invite/<token>

That creates a friend connection first, but does not route the recipient directly into the trip invite flow.

Desired behavior:

User receives:

“Hey! I planned a trip to Vail on BaseLodge. Join me here: [trip invite link]”

They tap the link.

If logged in:
- show trip invite/accept screen
- accept adds them to the trip

If not logged in:
- send them through signup/login
- preserve invite token
- after auth/onboarding, return them to the trip invite/accept screen
- accept adds them to the trip

==================================================
IMPLEMENTATION DIRECTION
==================================================

Create a true trip-specific external invite system.

Preferred simple approach:

Add a new TripInviteToken model/table.

Suggested fields:

- id
- token
- trip_id
- inviter_user_id
- created_at
- used_at nullable
- expires_at nullable
- is_active boolean default true

Token should be random, unguessable, and URL-safe.

Route:

GET /trip-invite/<token>

Behavior:

- validate token
- validate trip exists
- validate inviter owns or can invite to trip
- if user is not logged in:
  - store invite token in session
  - redirect to login/signup with return path
- if user is logged in:
  - show a simple trip invite landing/accept screen

Accept action:

POST /trip-invite/<token>/accept

Behavior:

- validate token
- validate logged-in user
- prevent owner from accepting their own invite
- prevent duplicate participant records
- if existing SkiTripParticipant exists:
  - update appropriately if invited/declined
- else:
  - create SkiTripParticipant for current user with accepted status
- optionally create friend connection if appropriate and existing friend-connection logic supports it safely
- mark token used only if token should be one-time use; otherwise leave active if this link can invite multiple people

IMPORTANT DECISION:

Recommend whether external trip invite links should be:

A. reusable link for multiple invitees
or
B. one-time link

For this product, reusable is probably better for “share via text” and “copy link,” but audit and recommend.
