#168C - PostHog User Property Backfill

What & Why

PostHog server delivery is now confirmed working.

Historical activation events were missed because POSTHOG_KEY was not configured when many existing users completed onboarding and activation actions.

We do NOT want to backfill historical events because that would distort timelines and funnels.

Instead, backfill current user state properties so PostHog accurately reflects what is true about each user today.

This is a one-time backfill.

---

## GOAL

Populate PostHog person properties for all existing users.

Examples:

* has_completed_signup
* has_completed_onboarding
* has_pass
* has_availability
* has_wishlist
* has_trip
* has_friend_connection
* has_generated_invite

Add:

* activation_backfilled = true
* activation_backfilled_at = current date

Do not create historical events.

Do not create timeline activity.

Do not modify existing PostHog event history.

---

* Data quality concerns

Audit:

---

## PHASE 2 — BACKFILL DESIGN

Use PostHog identify()/person-property updates only.

Do not call capture().

Do not generate:

* signup_completed
* onboarding_completed
* pass_added
* availability_added
* wishlist_added
* trip_created
* friend_connected
* invite_generated

No historical event recreation.

Only update person properties.

---

## PHASE 3 — SAFETY

The backfill must be:

* Idempotent
* Safe to rerun
* Non-destructive

Do not overwrite newer values with older values.

Use the user's existing PostHog distinct_id strategy.

Do not send:

* Email
* Name
* Phone
* Precise location
* Free-text notes

---

Recommend:

* Management command
* Admin-only route
* One-time script
* Background task

Identify the safest option.
