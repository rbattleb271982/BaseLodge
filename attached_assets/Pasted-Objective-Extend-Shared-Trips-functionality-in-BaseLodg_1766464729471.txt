Objective
Extend Shared Trips functionality in BaseLodge by:
Allowing organizers to invite additional friends to an existing Shared Trip
Clarifying how Shared Trips appear on Friend Profiles
Enforcing social and scale guardrails to prevent misuse
This is a UI + behavior extension on top of existing Shared Trip logic.
Do NOT add chat, messaging, payments, or new permission systems.
CORE PRINCIPLES (LOCKED)
Only the Organizer can invite additional friends
Eligibility rules apply to all invites (no cold invites)
Group size is intentionally limited
UI should feel calm, obvious, and non-performative
PART 1: INVITING ADDITIONAL FRIENDS TO A SHARED TRIP
UI ENTRY POINT
On a Shared Trip card (My Trips view):
Show secondary action:
“Invite friends”
Only visible to the Organizer
Must not visually compete with core trip details
INVITE FRIENDS FLOW
On tap “Invite friends”:
Open modal or bottom sheet
Show list of eligible friends only
Eligibility filtering (MANDATORY, INVISIBLE):
Include friends only if at least ONE is true:
Shared trip overlap exists
Shared open availability exists
Friend card shows “Trip invites: On”
Exclude:
Organizer
Already accepted participants
Already invited participants
Friends who declined twice
Do NOT show eligibility reasons in the UI.
FRIEND LIST DISPLAY (INSIDE MODAL)
Each friend row shows:
Name
Ski identity line (Skier · Passes · Skill)
No icons
No eligibility explanation
SELECTION & LIMITS
Multi-select allowed
Maximum of 10 total participants per Shared Trip
Including organizer
If organizer attempts to exceed limit:
Block action
Show alert:
“Shared trips are limited to 10 people. For larger groups, we’ll support this later.”
POST-INVITE BEHAVIOR
Create TripGuest records with:
status = invited
Invited users see:
“Invited to this trip”
Organizer sees status:
“Invited”
Decline rules:
First decline → may be re-invited once
Second decline → permanently excluded for that trip
No notifications beyond existing patterns.
PART 2: SHARED TRIPS ON FRIEND PROFILES (TIGHTENING)
FRIEND PROFILE — TRIPS SECTION
When displaying a friend’s trips:
Label Shared Trips clearly:
“Shared Trip”
Show trip dates
Do NOT show organizer name
Examples:
“Shared Trip · Jan 15–17”
“Shared Trip · Feb 2–5”
PARTICIPATION VISIBILITY
If you are invited:
You see “Invited” on your own view only
If you are accepted:
You see “Attending” on your own view only
Other friends:
See generic Shared Trip with dates only
This avoids social pressure.
TAP BEHAVIOR
Tapping a Shared Trip on a Friend Profile:
Navigate to Trip Detail page
Role-based actions:
Organizer: Invite friends
Accepted participant: Attending state
Invited user: Accept / Decline
Non-participant: Read-only view
No invite actions for non-participants.
PART 3: VISUAL & COPY RULES (LOCKED)
Terminology:
“Shared Trip”
“Organizer”
CTA: “Invite friends”
Tone:
Informational
Calm
No icons, emojis, or badges
EXPLICITLY OUT OF SCOPE
Do NOT build:
Messaging or chat
Cost splitting
RSVP counts beyond invited / accepted / declined
Editing participant lists beyond invite / re-invite
Multi-resort trips
Large group or event flows
ACCEPTANCE CRITERIA
Organizer can invite additional friends to a Shared Trip
Eligibility filtering works silently
Group size cannot exceed 10
Declined users can be re-invited once only
Shared Trips on Friend Profiles show dates but not organizer
Invite status is private
Role-based actions behave correctly
No regressions to solo trip behavior

Do not finish until all tests pass.
