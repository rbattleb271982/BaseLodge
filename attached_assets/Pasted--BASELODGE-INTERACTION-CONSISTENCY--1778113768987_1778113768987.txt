║ BASELODGE INTERACTION CONSISTENCY + CARD SYSTEM REFACTOR ║
╚════════════════════════════════════════════════════════════╝

We need to standardize interactive surfaces across the BaseLodge app and remove ambiguous inline navigation arrows.

GOAL

Move BaseLodge toward a cleaner Apple/editorial interaction model:

1. If an entire surface is tappable:
   - the surface itself should visually communicate interactivity
   - remove standalone inline arrows
   - keep only secondary utility actions like dismiss (`×`) if needed

2. Chevron-style navigation rows (`›`) should ONLY exist in traditional list/settings navigation patterns.

3. Clickable profile metadata and stats need stronger visual affordances.

────────────────────────────────────
IMPORTANT ARCHITECTURAL DECISION
────────────────────────────────────

The app should become selectively card-based, not universally card-based.

CARD-BASED SURFACES:
- Home feed modules
- Opportunities
- Happening/activity surfaces
- Invite modules
- Recommendation/social surfaces

These should feel warm, discoverable, editorial, and slightly elevated.

ROW-BASED SURFACES:
- Settings
- Profile navigation
- Mountains lists
- Friends lists
- Equipment rows
- Utility/drill-in navigation
- Trips index rows unless they are clearly recommendation/opportunity surfaces

These should remain cleaner, denser, and more structural using traditional row navigation patterns.

Do NOT convert the entire app into cards.
Maintain contrast between editorial/social surfaces and utility/navigation surfaces.

────────────────────────────────────
PART 1 — REMOVE REDUNDANT INLINE →
────────────────────────────────────

Audit all logged-in templates/components for:
- inline `→`
- SVG arrow icons
- row-end navigation arrows
- any arrow whose purpose is only “this is tappable”

Especially audit:
- Home feed/opportunities
- Invite modules
- Friends activity cards
- Trip opportunity modules
- Happening/activity surfaces
- Any remaining ideas/opportunity surfaces
- Cards with dismiss (`×`) actions

For tappable cards/surfaces:
- REMOVE the inline arrow entirely
- Preserve existing tap/click behavior
- Entire card/surface should remain tappable

DO NOT remove:
- traditional list-row chevrons in settings/profile navigation
- disclosure arrows for accordions
- back navigation arrows

────────────────────────────────────
PART 2 — CREATE A CONSISTENT CARD SYSTEM
────────────────────────────────────

For tappable feed/opportunity/invite/social surfaces only:

Convert them into visually distinct cards.

Card styling direction:
- editorial/premium
- subtle tonal separation
- NOT dashboard/SaaS styling
- inset with breathing room

Use:
- slightly different surface color from page background
- subtle border
- soft radius
- very soft/no shadow
- generous internal padding
- increased vertical spacing between cards

Design targets:
- feels like layered paper/surfaces
- obvious tap target
- clean mobile-first hierarchy
- warm and polished without feeling heavy

Suggested style direction:
- page background remains cream/off-white
- card background slightly lighter/warmer than the page background
- radius: ~18–22px
- subtle border using low-opacity maroon/charcoal
- very subtle active state with opacity + scale

DO NOT:
- introduce harsh gray cards
- use heavy shadows
- create “material dashboard” styling
- convert every row in the app into a card

────────────────────────────────────
PART 3 — SECONDARY ACTION CLEANUP
────────────────────────────────────

Dismiss (`×`) actions should become visually secondary.

Update dismiss actions:
- smaller visual weight
- slightly reduced opacity
- aligned upper-right where appropriate
- preserve accessibility/tap target
- dismiss should not visually compete with the card content

Dismiss behavior:
- Treat dismissals as algorithmic deprioritization, not necessarily permanent deletion.
- Preserve existing backend behavior if already implemented.
- Do not introduce destructive deletion unless already supported.
- If only frontend dismissal exists today, keep behavior lightweight and safe.

The card content should remain the primary focus.

────────────────────────────────────
PART 4 — CLICKABLE PROFILE METADATA
────────────────────────────────────

We need stronger visual affordances for clickable profile metadata.

On the profile/home identity card:

The following values are clickable and should visually look clickable:
- Skier / Snowboarder
- Skill level
- Pass type
- Actual skis value
- Actual boots value

Update these to:
- maroon text
- always-visible subtle underline
- preserve existing tap behavior
- route directly to the relevant edit destination
- maintain premium/editorial styling

DO NOT:
- style them like aggressive blue web hyperlinks
- add arrows
- add icons
- add intermediate screens

They should feel refined, intentional, and clearly tappable.

────────────────────────────────────
PART 5 — CLICKABLE STATS ROW
────────────────────────────────────

The profile/home stats row should also visually communicate interactivity.

Current stats:
- Upcoming Trips
- Mountains Visited
- Wishlist Mountains

Requirements:
- Each stat should be its own tappable area
- Each stat should route directly to its relevant destination
- Add subtle visual affordance that each stat is interactive
- Maintain current centered layout and hierarchy
- Avoid arrows/icons
- Preserve premium Apple/editorial feel

Potential direction:
- always-visible subtle underline on stat labels
- maroon or slightly darker interactive label treatment
- subtle opacity + scale on press
- maintain the large number / small label hierarchy

DO NOT make the entire stats row one single tap target.
Each stat should be independently tappable.

────────────────────────────────────
PART 6 — GLOBAL INTERACTION RULES
────────────────────────────────────

Establish these interaction conventions consistently:

SYSTEM A — TAPPABLE CARDS
Used for:
- feed modules
- opportunities
- invites
- passive activity cards
- recommendation/social surfaces

Behavior:
- entire surface tappable
- NO inline navigation arrow
- optional dismiss/utility action allowed
- subtle opacity + scale press state
- visually contained as a card

SYSTEM B — NAVIGATION ROWS
Used for:
- settings
- profile sections
- account rows
- equipment lists
- mountain lists
- friends lists
- drill-in rows

Behavior:
- keep chevron/disclosure affordance where appropriate
- full row tappable
- no competing dismiss actions
- remain row-based, not card-heavy

SYSTEM C — CLICKABLE METADATA
Used for:
- profile identity traits
- pass type
- skis
- boots
- stat surfaces

Behavior:
- maroon text
- always-visible subtle underline or refined affordance
- direct edit/detail destination
- no loud web-link aesthetics
- no arrows/icons

────────────────────────────────────
PART 7 — QA
────────────────────────────────────

Verify:
- no tap behavior regressed
- dismiss actions still work
- dismissals are safe and non-destructive unless already designed otherwise
- spacing consistent across cards
- card styling applies only to editorial/social/recommendation surfaces
- row styling remains intact for settings, profile navigation, mountains, friends, and equipment
- no arrows accidentally removed from true nav rows
- no disclosure arrows accidentally removed from accordions
- no back arrows removed
- no layout overflow on iPhone widths
- no broken alignment in bottom-nav safe area layouts
- all interactions remain mobile-first
- clickable metadata clearly reads as interactive
- each stat is independently tappable
- stats row interactions feel intentional and discoverable
- press states feel subtle and premium

Use checkpoints sparingly. Only use them if absolutely necessary.
