Investigate how Visited Mountains, Wishlist (Bucket List) Mountains, and all other resort selection flows are currently stored, queried, and rendered, and ensure there is a single source of truth for resorts across the app.

Context
Visited Mountains and Wishlist Mountains appear to be backed by separate tables or data paths. Additionally, there may be multiple places in the app where a user selects a resort, potentially using different tables or queries. Before making changes, we need a full diagnosis.

What to investigate
	1.	Resort source of truth

	•	Identify the canonical table/model that represents resorts (mountains).
	•	Confirm whether ALL resort selections reference this same table by ID.
	•	Call out any places where resort names or metadata are duplicated instead of referenced.

	2.	Visited vs Wishlist storage

	•	Identify how Visited Mountains are stored.
	•	Identify how Wishlist / Bucket List Mountains are stored.
	•	Confirm whether these reference the same Resort table or separate copies.
	•	Document whether the distinction is:
• separate tables
• a single join table with a status/type flag
• multiple join tables pointing to the same Resort table

	3.	All resort selection surfaces
Audit every place in the app where a user can choose a resort, including but not limited to:

	•	Adding a trip
	•	Editing a trip
	•	Marking a resort as visited
	•	Adding a resort to wishlist / bucket list
	•	Any onboarding or profile-related resort selection
	•	Any admin or debug tools that create or associate resorts

For each surface, confirm:
	•	Which table is queried
	•	Which table is written to
	•	Whether the resort ID comes from the canonical Resort table

	4.	Consistency and drift

	•	Identify any cases where:
• One feature uses a different resort table
• One feature stores resort names instead of IDs
• One feature joins differently than others
	•	Call out whether this separation is intentional or legacy drift.

What NOT to do yet
	•	Do not refactor schemas yet
	•	Do not merge tables yet
	•	Do not change UI behavior yet
	•	Do not migrate data yet

Deliverable

Provide a written summary that includes:
	•	The canonical Resort source of truth
	•	How Visited and Wishlist resorts are differentiated today
	•	A list of all resort-selection entry points and what they use
	•	Any inconsistencies or risks discovered
	•	A recommendation for how to unify or standardize resort selection if needed

Acceptance Criteria
	•	We clearly know which table is the single source of truth for resorts
	•	We know whether Visited, Wishlist, Trips, and other features all reference it
	•	We understand whether current separation is intentional or accidental
	•	We have enough clarity to decide on a clean unification plan next
