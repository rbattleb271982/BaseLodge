3. Fix availability overlap cards being suppressed when no shared wishlist resort exists

Problem:
Availability overlaps only show if the user and friend share a wishlist resort.
That is too strict.

Expected behavior:
If user and friend are both free on the same dates, surface an availability_overlap idea even if there is no shared resort.

Implementation:

- If no shared resort exists, set:

- Create a valid availability_overlap candidate with:

- Make sure templates/home.html handles resort_id=None gracefully.
- For resort_id=None cards, CTA should go to /add_trip, not a resort detail page.

4. Fix deduping for resort_id=None cards

Problem:
Current dedupe is one row per resort_id.
If resort_id is None, all no-resort availability cards collide into one key.

Fix:
In _try_add(candidate), use:

rid = candidate["resort_id"]
key = rid if rid is not None else ("no_resort", candidate.get("start_date"))

Then store candidates by this key.

This allows multiple no-resort availability overlap cards on different dates.

5. Improve sorting without changing the card model

Current sort:
friend_count desc, then earliest start_date.

Keep that primary behavior, but make it more deterministic and useful.

Use:
def _sort_key(r):
    days_away = (r["start_date"] - date.today()).days if r.get("start_date") else 999
    signal_rank = {
        "friend_trip": 1,
        "availability_overlap": 2,
        "wishlist_overlap": 3,
    }.get(r.get("idea_type"), 9)
    return (-r["friend_count"], days_away, signal_rank)

rows.sort(key=_sort_key)
