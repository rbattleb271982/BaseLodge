BaseLodge Final QA — Pass 1: Account Deletion, Ghost Users, and Social Graph Integrity

This is a READ-ONLY QA/testing pass.

Do not change code.
Do not delete data.
Do not create migrations.
Do not fix issues.
Only inspect, test, document, and report.

Purpose:
BaseLodge is preparing for Apple App Store and Google Play submission. This pass focuses on one of the highest-risk launch areas: account deletion and whether deleted users disappear cleanly from all social surfaces.

Test and inspect:

1. Account deletion flow
- Confirm a user can delete their account from inside the app.
- Confirm the deletion flow requires typing the email address manually.
- Confirm the delete button is not enabled by placeholder text.
- Confirm successful deletion logs the user out.
- Confirm the deleted user cannot log back in.
- Confirm failed deletion does not log the user out.
- Confirm clear success/error messaging.

2. Deleted-user propagation
After a test user is deleted, verify they no longer appear in:
- Friends lists
- Friend requests
- Pending invites
- Home feed / Happening
- Ideas / destination suggestions
- Trip overlap surfaces
- Mountain social proof
- Trip invite views
- Notifications/message-event surfaces shown to users
- Any profile/search surfaces
- Any admin dashboard user lists unless intentionally retained for audit

3. Friendship integrity
Inspect whether deletion removes or safely handles:
- friendships sent by the deleted user
- friendships received by the deleted user
- pending friend requests
- accepted friend relationships
- duplicate friend rows
- one-sided friendships
- orphaned friendships

4. Trip/social integrity
Inspect whether deletion safely handles:
- trips owned by the deleted user
- trip participants
- trip invites sent by the deleted user
- trip invites received by the deleted user
- friend activity generated by the deleted user
- visibility/feed items connected to the deleted user

5. Push/token cleanup
Inspect whether deletion safely handles:
- active push tokens
- stale push tokens
- OneSignal external user references if visible
- MessageEventLog rows involving the deleted user
- future push attempts to the deleted user

6. Re-signup test
If safe in the test environment:
- Create a test user.
- Connect them to another test user.
- Create at least one trip.
- Send/accept a friend request.
- Delete the test user.
- Re-sign up with the same email.
- Confirm the new account starts fresh and does not inherit old friends, trips, requests, push tokens, or social visibility.

7. Database integrity checks
Inspect relevant tables for orphaned rows involving deleted users:
- users
- friendships
- friend invitations
- trips
- trip participants
- trip guests
- trip invites
- push tokens
- message event logs
- activity logs
- any user profile/preferences tables

Report:
Return a structured report with:

A. Executive Summary
- PASS / FAIL / PARTIAL
- Is account deletion launch-ready?
- Does this block App Store / Google Play submission?

B. Critical Findings
For each issue:
- Severity
- User impact
- App Store / Google Play risk
- Steps to reproduce
- Suspected files/routes/models
- Recommended fix direction

C. Data Integrity Findings
List any orphaned rows, ghost users, one-sided friendships, stale push tokens, or deleted-user references.

D. Social Surface Findings
List every place a deleted user still appears.

E. Recommended Fix Order
Give a numbered fix order from highest to lowest priority.
