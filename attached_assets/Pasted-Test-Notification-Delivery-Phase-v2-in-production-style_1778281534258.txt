Test Notification Delivery Phase v2 in production-style app testing.

Do not implement new features unless a small fix is required to complete the test safely.

Goal:
Verify that the first real user-facing OneSignal push delivery works for:
1. friend.request.accepted
2. trip.invite.created

Before testing:
- Confirm the app starts cleanly
- Confirm the latest code is running
- Confirm /admin/message-events loads
- Confirm OneSignal env vars are present if available

Test path 1: friend.request.accepted
- Use two test users
- User A sends a friend request to User B
- User B accepts the friend request
- Verify existing Activity behavior still works
- Check /admin/message-events
- Confirm there is one friend.request.accepted row for User A
- Confirm:
  - channel = push
  - provider = onesignal
  - delivery_status = sent, skipped, or failed
  - suppression_reason is only present if skipped
  - payload includes event, user_id, deep_link, and screen

Test path 2: trip.invite.created
- Use two test users
- User A creates or edits a trip and invites User B
- Verify existing trip invite behavior still works
- Check /admin/message-events
- Confirm there is one trip.invite.created row for User B
- Confirm:
  - channel = push
  - provider = onesignal
  - delivery_status = sent, skipped, or failed
  - suppression_reason is only present if skipped
  - payload includes event, trip_id, deep_link, and screen

Dedupe check:
- Try to repeat the same invite action within the dedupe window
- Confirm duplicate behavior does not send a second push
- Confirm any duplicate audit row, if created, is SKIPPED / DUPLICATE_EVENT

Failure safety check:
- Confirm push failures do not break the user action
- Friend accept should still complete
- Trip invite should still complete
- Activity behavior should remain unchanged

Do not wire any other events.
Do not change schema.
Do not change APNs/FCM direct behavior.
Keep APNs/FCM direct sends admin/test-only.

At the end, report:
- Test users used
- Whether each flow passed
- MessageEventLog rows created
- Delivery statuses observed
- Any push received on device, if testable
- Any bugs found
- Whether this is safe to keep deployed

Use checkpoints sparingly.
