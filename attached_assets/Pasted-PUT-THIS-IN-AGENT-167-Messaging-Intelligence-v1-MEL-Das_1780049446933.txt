#167 - Messaging Intelligence v1 — MEL Dashboard

What & Why

The existing /admin/messaging dashboard covers invite-link and trip-invite flows using the InviteToken, Invitation, Friend, and PushDeviceToken tables — but it does not query MessageEventLog at all.

The MEL has been recording every push event (friend requests, trip invites, automations, test/broadcast) with full delivery status, suppression reasons, and OneSignal provider IDs since Phase C shipped.

This task surfaces that data in the dashboard so you can see push volume by time window, event-type leaderboard, delivery health breakdown, and suppression analysis without leaving /admin/messaging.

Done looks like

* A new "MEL Volume" row group in the dashboard showing push messages L7 / L30 / L90 (total, sent, skipped, failed)
* An "Event Type Activity" leaderboard: the most-triggered event_name values ranked by volume, with status breakdown per type
* A "Delivery Health" funnel using MEL delivery_status counts (sent / skipped / failed) with % rates
* A "Suppression Breakdown" list showing which suppression_reason values are most common (channel_unavailable, user_opted_out, duplicate_event, etc.)
* An updated dynamic insight that can reference MEL data (e.g. "friend.request.created accounts for 48% of all push activity")
* Updated "Messaging Status" block: tracked_signals and reliable_signals counts reflect the completed audit (9 GREEN, 4 YELLOW, 7 RED)
* All new sections are mobile-friendly, extend the existing .ms-* CSS namespace, and render gracefully when MEL has 0 rows

Out of scope

* Push open / click tracking (RED — requires OneSignal webhooks, not in this task)
* Email messaging (RED — EmailLog inactive, SendGrid not wired into MEL)
* Marketing/broadcast MEL category (no events registered under Category.MARKETING)
* OneSignal subscription metadata per user (would require OneSignal API calls)
* New models, migrations, or new admin routes
* Anything requiring a new vendor or external API

Implementation Requirements

Do not create:
