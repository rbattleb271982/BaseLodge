Goal
Unify the profile card experience so that:
Your profile and friend profiles display the exact same information
Friend profiles are read-only
There is one shared profile card component
No data is hidden or reduced on friend profiles
This is a UI + permissions refactor only.
Do not add new fields or backend logic beyond what already exists.
1️⃣ SHARED PROFILE CARD (SINGLE SOURCE OF TRUTH)
Requirement
Create or refactor to a single reusable profile card component that is used for:
Logged-in user profile
Friend profile view
The component must accept:
user (the profile being viewed)
is_owner (boolean: viewing own profile vs friend)
2️⃣ PROFILE CARD CONTENT (SHOW IN ALL CASES)
Identity Section
Show the following for both own and friend profiles:
Rider Type
Skill Level
Pass type(s)
Primary Riding Style
Equipment:
Equipment status (own / rent / both)
Skis / Board brand + model (if present)
Boots brand + model (if present)
Display rules:
Only show fields that exist
No placeholders
No “Unknown”
Same order and formatting everywhere
3️⃣ PROFILE STATS ROW (SAME FOR ALL PROFILES)
Stats (exactly three)
Trips Created
Visited Mountains
Bucket List Mountains
Layout
3 columns
Label on top
Number underneath
No icons
Center-aligned
Interaction
Entire column tappable
Routing
Trips Created → user’s Trips list
Visited Mountains → user’s Visited Mountains list
Bucket List Mountains → user’s Bucket List list
On friend profiles, all destinations are read-only.
4️⃣ EDIT PERMISSIONS (ONLY DIFFERENCE)
If is_owner == true
Show edit affordances where they already exist
Allow add/remove/edit actions
If is_owner == false
Hide all edit controls
Disable add/remove actions
Lists open in read-only mode
No “Edit”, “Add”, “Remove”, or inline inputs
This must be enforced at the UI level.
5️⃣ WHAT NOT TO DO (IMPORTANT)
❌ Do not create a second “friend profile card”
❌ Do not hide equipment, skill level, or pass info on friend profiles
❌ Do not add privacy layers or new toggles
❌ Do not add new backend fields
❌ Do not change data models
This is one component, two permission states.
6️⃣ DONE CRITERIA
Friend profiles visually match the user’s own profile card
Only difference is editability
Stats row behaves identically
All taps work correctly in read-only mode
No duplicated templates or logic
