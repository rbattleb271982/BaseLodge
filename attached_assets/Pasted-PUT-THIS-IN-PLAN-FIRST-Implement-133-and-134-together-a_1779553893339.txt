TASK #134 — INVITE AUTH + REMOVE FRIEND REGRESSION
==================================================

WHAT & WHY

Two functional fixes for the TestFlight/native-app experience.

1. Invite-link auth persistence
Authenticated TestFlight users opening friend/trip invite links from:
- iMessage
- Gmail
- Apple Mail
- email apps

should remain fully inside the app without unnecessary Safari redirects or login prompts.

Current issue:
- "Continue in Browser" renders unconditionally even for authenticated native users.

2. Remove Friend regression
Removing a friend may still leave ghost connected states because:
- Invitation rows with status='accepted' survive removal
- some surfaces may incorrectly derive connected-state from Invitation.status instead of Friend row existence
- cached profile/friendship state may persist after redirect

==================================================
DONE LOOKS LIKE
==================================================

AUTHENTICATED INVITE FLOW
- User is logged into TestFlight app
- Invite opened from iMessage/email
- App opens directly
- Confirmation screen appears inside app
- NO "Continue in Browser" button
- NO login prompt
- NO Safari redirect
- Friendship/trip invite resolves immediately

UNAUTHENTICATED FLOW
- Existing auth behavior preserved
- invite token stored in Flask session
- invite resolves after login/signup
- "Continue in Browser" remains visible

REMOVE FRIEND
- User A removes User B
- Both disappear from each other’s Friends lists
- No connected-state remains anywhere
- No shared overlaps remain
- No ghost friendship state survives
- Re-friending later works correctly
