2. Friends-by-date content

This is the main body of the page.

Trip filtering:

* Include only:

  * upcoming trips (start_date > today)
  * in-progress trips (start_date <= today <= end_date)
* Exclude past trips (end_date < today)
* Use canonical trip logic (get_trip_status) to validate

Query rules:

* Query SkiTrip where resort_id == resort.id
* Include owner + accepted participants
* Only include:

  * current user
  * friends of current user

Grouping:

* Group by (start_date, end_date)
* Sort groups chronologically by start_date
* Do NOT create empty groups

Within each group:

* Show date range on the left
* Show nights count on the right (end_date - start_date)

---

3. Row structure

Each row represents one real person tied to a real trip.

Line 1:

* Full name on the left (or “You” for current user)
* Status text on the right

Line 2:

* Rider type · Skill level · Pass

Row rules:

* Show “You” instead of the user’s actual name
* If “You” is present in a group, always place “You” first
* After that, sort alphabetically by full name
* Use full first + last names for friends
* A friend can appear in multiple groups (do NOT dedupe across groups)

Pass formatting:

* Normalize pass_type to main pass family only

  * e.g. "Ikon Base+", "Ikon Pass" → "Ikon"
* If no pass → show “No pass”

Identity line formatting:

* Format exactly as:
  [rider_type] · [skill_level if present] · [pass]

Rules:

* If skill_level is null → omit it cleanly (no extra separator)
* If pass is null → show “No pass”
* No malformed separators (no double dots or trailing dots)

Status labels:

* “Going” for accepted
* “Considering” for invited/pending (if surfaced in current logic)
* Do NOT use any other status labels

Status design:

* Plain right-aligned text only
* No pills
* No heavy styling
