Goal  
Implement a v1 Friends → Updates activity feed that shows recent activity from direct friends only.  
This is an ongoing activity feed, not a notification system — there is no unread state, no badges, no alerts.

Navigation (Do Not Change Existing Nav)
- The bottom navigation already exists and includes:
  Trips · Friends · Invite · Profile
- Do NOT modify the bottom navigation.
- The activity feed lives entirely inside the Friends screen.
- The Invite tab remains separate and unchanged.

Friends Screen Layout
- Inside the Friends screen, add a segmented control at the top:
  [ Updates ]   Friends
- Updates is the default (left) tab when the Friends screen loads.
- Friends tab shows the existing friends list exactly as it does today (no changes).

Updates Tab
- Displays a reverse-chronological feed of friend activity.
- Ongoing feed only:
  - No unread/read state
  - No dots, badges, or counts
  - No alerting behavior
- Neutral copy tone.
- No emojis.
- Activity items disappear automatically if the underlying object is deleted.

Empty State
If no activity exists, show:
No updates yet
When friends add or update trips, you’ll see it here.

Activity Types (v1 Only)
Create activity entries only for:
- Friend adds a new trip
- Friend updates trip dates
- Friend joins a trip you’re on
- Friend accepts your trip invite
- Friend accepts your connection invite
- Friend’s trip overlaps with one of yours

Do NOT create activity for:
- Profile edits
- Equipment updates
- Pass changes
- Friend removals
- Pending invitations

Tap Behavior
Always navigate to the primary object of the activity.

- Trip added → Trip detail page
- Trip updated → Trip detail page
- Friend joins your trip → Trip detail page
- Friend accepts trip invite → Trip detail page
- Friend accepts connection invite → Friend profile page
- Trip overlap detected → Trip detail page

Data Model
Create a new model:

Activity
- id (primary key)
- actor_user_id (foreign key → User)
- recipient_user_id (foreign key → User)
- type (string enum)
- object_type (string: "trip" | "user")
- object_id (integer)
- created_at (datetime)

Rules:
- No read_at field
- Store activity forever
- Delete activity records if the underlying object is deleted

Friend Resolution (Authoritative)
Direct friends must be determined exclusively using the existing Friend model, which represents a mutual, accepted connection.

Rules:
- A user’s direct friends are defined only by the Friend table.
- Friendship is bidirectional:
  - Friend.user_id = current_user.id
  - OR Friend.friend_id = current_user.id
- The “other” user in the row is the friend.
- Do NOT infer friendship from Invitation records.

Conceptual example (not prescriptive):
friend_ids = (
    query Friend.friend_id where Friend.user_id == current_user.id
    UNION
    query Friend.user_id where Friend.friend_id == current_user.id
)

Only activities where actor_user_id is in this resolved friend set should appear in Updates.

Backend Logic
- Create Activity records at the moment each supported event occurs.
- Only create activity for direct friends.
- Do not create activity for the acting user themselves.
- Query activities where:
  - recipient_user_id = current_user.id
  - actor_user_id is a direct friend
- Order by created_at DESC.

Frontend Rendering
Each activity row must show:
- Friend name
- Short neutral action text
- Context (trip name if applicable)
- Relative timestamp (e.g., 2h ago, Yesterday)

No icons, no emojis, no highlight states.

Constraints
- Do not add unread/read logic
- Do not add badges or alert indicators
- Do not add push notifications
- Do not modify the Invite tab
- Do not refactor unrelated models, routes, or screens

Completion Criteria
- Friends → Updates loads correctly
- Activity feed displays correct items
- Navigation behaves exactly as specified
- Empty state appears when appropriate
- No visual alerting exists
