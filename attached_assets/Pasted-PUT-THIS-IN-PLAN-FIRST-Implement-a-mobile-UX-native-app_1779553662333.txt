PART 2 — INVITE LINK AUTH PERSISTENCE + NATIVE FLOW
==================================================

Context:
Authenticated TestFlight users should never be forced through unnecessary browser/login flows when opening invite links from iMessage/email.

Currently:
- invite landing pages may show “Continue in Browser” even for authenticated native users
- Safari redirects fragment the session
- the flow feels webby instead of native

Goal:
If the app is installed AND the user is already authenticated:
- invite links should resolve directly inside the app
- preserve the authenticated session
- avoid unnecessary auth prompts
- complete the invite flow seamlessly

==================================================
AUTHENTICATED NATIVE USER FLOW
==================================================

Desired behavior:
- User taps invite link from iMessage/email
- BaseLodge app opens directly
- Existing authenticated session is reused
- Invite confirmation screen appears inside the app
- “Continue in Browser” is hidden
- User taps Connect/Accept
- Friendship/trip invite resolves immediately
- User routes to Friends tab or Trip Detail

No:
- login prompts
- Safari detours
- session loss
- redundant auth redirects

==================================================
UNAUTHENTICATED FLOW
==================================================

Existing behavior should remain:
- token stored in Flask session
- user routed through auth
- invite resolved after login/signup/reset-password

“Continue in Browser” should remain visible for unauthenticated users only.

==================================================
IMPLEMENTATION REQUIREMENTS
==================================================

Audit:
- templates/invite_landing.html
- templates/trip_invite_token_landing.html
- invite token/session handling
- Capacitor deep-link handling
- Flask session continuity
- SameSite/Secure cookie behavior
- login redirect logic
- existing invite_token session storage
- app-open detection logic

Important:
- Do NOT break existing web invite flows.
- Native authenticated flow should be additive, not a replacement.

Potential fixes:
- hide “Continue in Browser” for authenticated native users
- preserve in-app session continuity
- improve native-shell invite hydration
- ensure authenticated users are not treated as anonymous
- avoid unnecessary auth checks before checking current session state

Session audit:
- Verify whether SESSION_COOKIE_SAMESITE or redirect behavior is causing session loss.
- If SESSION_COOKIE_SAMESITE=Strict is confirmed problematic, consider Lax.
- Only change config if regression is confirmed.

Additional QA:

PART 4 — REMOVE FRIEND REGRESSION FIX
==================================================

Context:
Removing a friend does not fully sever the friendship across all app surfaces.

Likely causes:
- Invitation rows with status='accepted' surviving removal
- stale cached connected-state UI
- surfaces deriving connected-state incorrectly

==================================================
REQUIRED BEHAVIOR
==================================================

Removing a friend must fully disconnect both users.

After removal:
Users should NOT appear in:
- Friends lists
- Connected-state profile UI
- Shared overlap sections
- Invite-connected pickers
- Friend-based recommendation logic
- Friends-going social logic

Users SHOULD:
- be able to re-send a friend request later

==================================================
IMPLEMENTATION REQUIREMENTS
==================================================

Audit:
- remove_friend API
- remove_friend_web
- all connected-state derivations
- Invitation.status usage
- Friend row existence checks
- client-side friend caches
- overlap payload hydration
- stale optimistic UI

If Invitation.status='accepted' is used anywhere as a connected-state signal:
- update removal flows to set:
  - removed
  - cancelled
  - or equivalent non-connected state

Ensure:
- both mirrored Friend rows removed/deactivated
- cached state invalidated
- UI refreshes after removal
- redirects land on fresh Friends list state

Additional audit:
- Verify client-side caches/store objects do not retain stale friendship state.

Relevant files:
- app.py remove_friend routes
- models.py Friend/Invitation
- friend profile routes/templates

Acceptance criteria:
- User A removes User B
- Both users disappear from each other’s Friends lists
- No shared overlap sections remain
- Profiles no longer show connected state
- Re-friending works correctly
- No ghost connected state anywhere
