We are making a cohesive update to Profile-related edit flows in BaseLodge to improve clarity, reduce friction, and ensure users always explicitly save changes.
This work includes:
Equipment ownership logic
Brand + Model support
Primary vs secondary setups
Explicit Save buttons for edit screens
Backward compatibility for existing users
All changes must be low-friction, explicit, and consistent across the app.
OVERALL OBJECTIVES
Let users accurately describe their equipment (Brand + Model)
Avoid turning BaseLodge into a gear catalog
Only show inputs that are relevant to the user’s choices
Ensure all profile edits have an explicit Save action
Preserve all existing user data without forcing updates
PART 1 — Equipment Ownership & Setup Flow (Authoritative)
1️⃣ Equipment ownership selection (copy + logic)
Replace existing copy:
❌ “Have own equipment”
❌ “Needs rentals”
With:
✅ “I have my own equipment”
✅ “I use rentals”
This selection controls which fields are shown.
2️⃣ Ownership-driven UI behavior
If user selects “I have my own equipment”:
Show Primary Equipment Setup
Primary setup includes:
Brand (dropdown)
Model (free-text)
Boots: Brand + Model
Primary setup is required only when equipment is owned
Secondary setup:
Do NOT show by default
Show a button: “+ Add another setup”
Tapping reveals a secondary setup with the same fields
Secondary setup is optional
If user selects “I use rentals”:
Do NOT show:
Primary setup
Secondary setup
Do NOT require equipment details
Preserve previously entered equipment data but hide it
Switching back to “I have my own equipment” restores prior inputs

5️⃣ Data storage rules
Store Brand and Model as separate database fields
Model fields must be nullable
Do NOT concatenate Brand + Model
Apply consistently to:
Skis / Snowboards
Boots
6️⃣ Display logic (everywhere in the app)
Brand + Model → Brand Model
Brand only → Brand
Model only → Model
Neither → do not render equipment
Never display “Unknown model”.
7️⃣ Backward compatibility (critical)
Existing users must not be forced to update
Existing brand-only setups must continue to render correctly
Switching ownership types must not delete data
Do NOT block saving due to missing model data
PART 3 — Explicit Save Buttons for Profile Edit Screens
Several profile edit screens allow changes without an explicit Save action. This is a UX bug and must be fixed.
8️⃣ Wishlist Destinations — Add Save button
Add a primary “Save” button
Prefer sticky bottom placement
Save button behavior:
Disabled when no changes are made
Enabled when selection differs from saved state
On Save:
Persist wishlist selections
Navigate back
No confirmation modal
This screen is NOT autosave
9️⃣ Mountains Visited — Add Save button (NEW)
Apply the same Save behavior as Wishlist Destinations:
Add a primary “Save” button
Sticky bottom placement preferred
Disabled until changes are made
On Save:
Persist selected mountains
Navigate back
No confirmation modal
10️⃣ Explicitly do NOT
Do NOT auto-save silently
Do NOT rely on back navigation to persist changes
Do NOT add confirmation dialogs
Do NOT introduce new button styles
Do NOT permanently expand card height due to edit states
ACCEPTANCE CRITERIA
Equipment inputs only appear when relevant
Users can add one or multiple equipment setups intentionally
Brand + Model display correctly everywhere
Wishlist Destinations has a Save button
Mountains Visited has a Save button
Changes persist only after Save
No regressions for existing users
