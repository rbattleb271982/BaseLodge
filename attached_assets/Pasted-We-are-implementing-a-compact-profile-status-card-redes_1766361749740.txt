We are implementing a compact profile/status card redesign across BaseLodge and updating the Equipment edit page to support Brand + Model.
This work is primarily UI/layout, with minimal, backward-compatible data handling.
No breaking schema changes. No forced user updates.
PART 1 — Profile / Status Card Redesign
Objective
Create a compact, horizontal, highly scannable profile card used across:
Home
Friends list
Friend Profile
The card should:
Minimize vertical height
Work well in lists
Clearly communicate identity, activity, and experience
Feel calm and efficient (Strava-like, not résumé-like)
Card Layout (Authoritative)
Row 1 — Header (single line)
Name (left, prominent)
Status indicator (right, inline, compact)
Examples:
“Starts today”
“Next trip · Jan 12–15”
“No upcoming trips”
Rules:
Status must NOT create extra vertical height
No banners or stacked status blocks
Row 2 — Identity (single line)
Combine into one horizontal line:
Home State only (state abbreviation)
Rider type
Pass type(s)
Skill level
Example:
CO · Skier · Epic & Ikon · Advanced
Rules:
Do NOT include mountain names here
Home mountain does NOT appear in the card
Row 3 — Equipment (optional, compact)
Equipment must support Brand + Model.
Display rules:
Single inline row
Smaller, lighter text
Must NOT wrap or add height
Format:
Skis: Brand Model
Board: Brand Model
Boots: Brand Model
Example:
Skis: Elan Ripstick · Boots: Nordica Speedmachine
Rules:
If model is missing, show brand only
If no equipment exists, do not render this row
Row 4 — Stats Row (structured, horizontal)
Create a horizontal stat group with three stat blocks, separated by thin vertical dividers.
Stat blocks (left → right):
Trips Planned
Small label
Larger number
Not expandable
Mountains Visited
Small label
Larger number
Expandable on tap
Wishlist Mountains
Small label
Larger number
Expandable on tap
Divider rules:
Thin vertical line between stat blocks
Light neutral color
No divider after the last stat
Interaction Rules
Mountains Visited and Wishlist Mountains:
Display numeric summary only
Expand on tap to show lists
Use identical behavior
Expansion must NOT permanently increase card height
Home Mountain Handling
Home Mountain is optional
Display only on the Profile screen
Do NOT display in compact cards
If not set, render nothing (no placeholders)
Spacing & Typography Guardrails
Favor horizontal layouts over vertical stacking
Reduce padding and vertical gaps
Use slightly larger text for readability
Multiple cards should fit comfortably on one mobile screen
Avoid tall, airy, or sectioned card designs
PART 2 — Equipment Edit Page (ADD MODEL SUPPORT)
Objective
Allow users to optionally specify an equipment model in addition to brand, so the app can display accurate gear (e.g., “Elan Ripstick”).
This must be backward-compatible.
Equipment Data Rules (Canonical)
For each equipment category (Skis / Board / Boots):
Brand — primary value
Model — optional
Examples:
Skis: Elan Ripstick
Boots: Nordica Speedmachine
Board: Burton Custom
UI Changes — Equipment Page
For each equipment type:
Brand
Existing behavior remains (dropdown or text input)
Model (NEW)
Optional text input
Placeholder example:
“e.g. Ripstick 96”
Not required to save
Data Handling Rules
Existing users with brand-only equipment:
Must continue to work unchanged
Display logic everywhere:
If model exists → show Brand + Model
If model is empty → show Brand only
What NOT to Do
Do NOT require model entry
Do NOT force users to update existing equipment
Do NOT introduce new tables unless strictly necessary
Do NOT normalize or validate models yet
Guardrails (Non-Negotiable)
Do NOT change backend business logic
Do NOT remove any existing data
Do NOT add new features beyond what’s specified
Keep all changes backward-compatible
Deliverables
Compact, horizontal profile/status card
State-only location in identity row
Structured stat row with vertical dividers
Expandable Mountains Visited + Wishlist
Equipment displayed as Brand + Model when available
Equipment edit page updated with optional Model field
No regressions
Proceed with implementation.
