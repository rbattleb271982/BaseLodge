==================================================
TEST 1 — REMOVE FRIEND SEVERS FRIEND ROWS AND ACCEPTED INVITATIONS
==================================================

Test name:
test_remove_friend_severs_friend_rows_and_accepted_invitation

Setup:
- Create User A and User B.
- Create bidirectional Friend rows.
- Create an Invitation row between the users with status="accepted".

Action:
- Call the canonical remove-friend logic, or POST to the web/API remove-friend route as authenticated User A.

Assertions:
- Both Friend rows are gone.
- The accepted Invitation row is no longer accepted.
- Expected final Invitation status should be "cancelled" unless the app uses another explicit non-connected status from the recent implementation.
- The test should cover both directions where applicable.

Purpose:
Prevent ghost connected-state regressions caused by surviving accepted Invitation rows.

==================================================
TEST 2 — RENTAL EQUIPMENT PERSISTS TO USER.EQUIPMENT_STATUS
==================================================

Test name:
test_rental_equipment_persists_to_user_equipment_status

Setup:
- Create a test User.
- Authenticate as that user.

Action:
- POST to the onboarding equipment route with the equivalent of:
  equipment_choice=rent

Assertions:
- user.equipment_status == "needs_rentals"
- The persisted User row reflects the rental state after the request.

Purpose:
Prevent regression where rental users are shown as having no equipment.

==================================================
TEST 3 — PASS DISPLAY OVERRIDE DOES NOT MUTATE NORMALIZE_PASS
==================================================

Test name:
test_pass_display_override_does_not_mutate_normalize_pass

Setup:
- Import normalize_pass from the canonical pass utility/module.
- Choose one overridden resort/pass ecosystem example, such as Killington or Palisades.

Action:
- Capture normalize_pass(raw_value) before get_all_active_resorts_map() / _pass_data override path runs.
- Run the resort map builder path with mocked/minimal data if needed.
- Capture normalize_pass(raw_value) again.

Assertions:
- normalize_pass(raw_value) returns the same value before and after the display override code runs.
- normalize_pass(raw_value) does NOT return a display override label like "Ikon".
- The display override remains display-only and does not mutate normalization behavior.

Purpose:
Ensure branded pass labels stay presentation-only and never contaminate pass normalization/storage logic.

==================================================
TEST 4 — HOME EMPTY OPPORTUNITY CARD ONLY WHEN FEED IS EMPTY
==================================================

Test name:
test_empty_opportunity_card_only_when_feed_empty

Setup:
- Create and authenticate a test User.
- Use Flask test client.

Action A:
- GET home with no generated opportunity/destination feed content.

Assertion A:
- Response HTML contains the empty-state card marker/class:
  bl-opp-empty-card

Action B:
- Seed or mock one real opportunity/destination feed item for that user.
- GET home again.

Assertion B:
- Response HTML does NOT contain:
  bl-opp-empty-card

Purpose:
Prevent regressions where the empty-state card appears when ideas exist, or disappears when the feed is genuinely empty.

If the project does not have an Opportunity model/table:
- mock or patch the feed builder used by the Home route so dest_feed=[] and dest_feed=[one item] can be tested deterministically.
- Do not invent a new production model.
