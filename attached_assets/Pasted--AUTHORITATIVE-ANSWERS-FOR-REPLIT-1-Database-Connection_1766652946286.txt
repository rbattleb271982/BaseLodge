1. Database Connections (DEV vs PROD)

Answer: Do NOT prompt for a production DATABASE_URL.

Decision:
	•	Production DB credentials should be provided via environment variables, not user input.
	•	The admin page should never ask for or accept a DB URL.

Implementation rule:
	•	DEV environment uses its existing DATABASE_URL
	•	PROD sync should be executed via a controlled mechanism, not by connecting directly from DEV UI to PROD DB

Rationale:
	•	Prompting for a PROD DB URL is unsafe
	•	Easy to mis-paste credentials
	•	Violates least-privilege principles
	•	Makes accidental prod writes far more likely

✅ The UI should never handle credentials.

⸻

2. Sync Mechanism (MOST IMPORTANT QUESTION)

Answer: Use the canonical export + existing prod sync endpoint.

👉 Choose option 2:

Export to canonical_resorts.json and rely on the existing
/admin/sync-resorts-from-canonical endpoint on PROD.

DO NOT write directly to the production database from DEV.

Correct flow:
	1.	Admin edits resorts in DEV
	2.	DEV exports a clean canonical_resorts.json
	3.	Admin explicitly triggers sync
	4.	PROD ingests that file via its existing sync endpoint

Why this is the right choice:
	•	You already have this mechanism
	•	Keeps environments isolated
	•	Allows review / rollback
	•	Matches how you’re treating resorts as a source-of-truth dataset
	•	Avoids cross-environment DB coupling (huge win)

✅ This also makes future CI / audits easier.

⸻

3. Resort Model Changes (Soft Delete vs Hard Delete)

Answer: YES — add an is_active boolean to Resort.

Decision:
	•	Add is_active = true (default)
	•	Use soft deletes for merges and removals

Rules:
	•	Merged / deprecated resorts → is_active = false
	•	User-facing lists filter on is_active = true
	•	Admin page can show inactive resorts if needed

Why this matters:
	•	Hard deletes are risky during merges
	•	Soft deletes allow:
	•	Rollback
	•	Audit
	•	Safer foreign-key handling
	•	This is a one-time, low-risk schema improvement

✅ This is worth doing now.

⸻

4. FK References During Merge

Answer: YES — all references must be repointed to the canonical resort.

When merging duplicates, update every place a resort is referenced:

Required updates:
	•	✅ User.visited_resort_ids (JSON)
	•	✅ User.wishlist_resort_ids (JSON)
	•	✅ User.home_resort_id (FK)
	•	✅ SkiTrip.resort_id (FK)

Merge behavior must be:
	•	Atomic (all-or-nothing)
	•	Safe (no orphaned references)
	•	Deterministic (no guessing)

After merge:
	•	Canonical resort remains active
	•	Duplicates marked is_active = false
	•	No user data lost

❗️Leaving old references in place is not acceptable.
