Please implement reliable invite share tracking for BaseLodge admin metrics.

Current State

BaseLodge currently has invite tokens and invite acceptance tracking, but it does not track actual outbound share intent. The current admin "Invites Today" metric counts internal Invitation rows + SkiTripParticipant INVITED rows, not copy/text/share actions.

Goal

Add an "Invites Sent Today" metric based on user share intent actions:

* Friend invite link copied
* Friend invite opened via SMS/text
* Trip invite link copied
* Trip invite opened via native share sheet
* Trip invite copied

Important:

This tracks share intent, not guaranteed message delivery. SMS and native share sheets cannot confirm the user actually sent the message.

Implementation Preference

Add a small local database table so the admin panel can query this directly.

New Model

InviteShareEvent

Fields:

* id
* user_id
* token_type: "friend" or "trip"
* token_id: nullable integer
* token: nullable string fallback
* action: "copy" | "text" | "share_sheet"
* source: "invite_page" | "friends_empty_state" | "trip_detail"
* created_at
* user_agent nullable
* platform nullable if easy to capture

Backend Endpoint

Add:

POST /api/invite/share

Payload:

{
"token_type": "friend" | "trip",
"token": "...",
"action": "copy" | "text" | "share_sheet",
"source": "invite_page" | "friends_empty_state" | "trip_detail"
}

Behavior:

* Require login.
* Validate action, source, and token_type.
* Look up the token when possible:

  * InviteToken for friend token_type
  * TripInviteToken for trip token_type
* Insert an InviteShareEvent row.
* Also fire a PostHog/server analytics event if the existing analytics helper pattern supports it.
* Return JSON success.
* Do not block the frontend share/copy UX if the tracking request fails.

Frontend Changes

1. templates/invite.html

Track:

* Copy invite link
* Invite via text

2. templates/friends.html empty state

Track:

* Copy invite link
* Invite via text

3. templates/trip_detail.html

Track:

* blShareTrip()
* blCopyInviteLink()

Important Frontend Behavior

* For copy actions, fire tracking before or after clipboard copy, but do not prevent copy if tracking fails.
* For SMS/text links, add an onclick handler so tracking fires, then opens the sms: URL.
* For navigator.share(), track "share_sheet" when the share sheet successfully completes.
* If AbortError is caught, do not count it.
* If browser limitations make this unreliable, document the exact behavior in comments.
* Tracking should be completely invisible to the user and never interfere with sharing.

Founder Push Notifications

Also add founder-only push notifications whenever an InviteShareEvent is created.

Requirement:

When any InviteShareEvent is successfully recorded, send a founder-only push notification using the same pattern as existing founder notifications (new user, app open, etc.).

Push Title

Invite Sent

Push Body Examples

* John Smith sent a friend invite · copy link
* John Smith sent a friend invite · text
* John Smith sent a trip invite · share sheet
* Someone sent a friend invite · copy link

Optional Enhancements

If easy:

* Include state abbreviation:

  * John Smith sent a friend invite · CO · text
* Include source:

  * John Smith sent a trip invite · trip detail · share sheet

Rules

* Only notify founder/admin recipients already used by existing founder push logic.
* Do not notify normal users.
* Do not block invite tracking if push sending fails.
* Send push only after InviteShareEvent commit succeeds.
* Wrap push sending in try/except with logging.
* Reuse existing founder push helper functions if available.
* If easy, prevent accidental duplicate pushes from rapid double-taps using an existing dedupe pattern. Otherwise document the behavior.
