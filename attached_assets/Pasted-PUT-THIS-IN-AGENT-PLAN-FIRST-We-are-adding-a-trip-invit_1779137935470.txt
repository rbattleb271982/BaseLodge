GOAL
==================================================

On the Trip Detail page, users should be able to invite people to a trip.

Keep the existing informational card:

“Friends heading to [Mountain]”

Then add a separate action directly beneath it:

“Invite friends to this trip →”

This keeps:

- “Friends heading to [Mountain]” = who else is going
- “Invite friends to this trip” = bring someone to this trip

- same width as existing cards
- rounded card style
- lightweight, not huge
- maroon text/accent
- right chevron
- consistent with BaseLodge card/list design

Do not make it a floating button.

FLOW A — EXISTING BASELODGE FRIENDS
==================================================

When tapped, open an invite modal or bottom sheet.

Section:

Friends on BaseLodge

Show connected friends only.

Exclude:

- current trip owner
- friends already participating in the trip
- friends already invited to this trip

User can select one or more friends.

On submit:

- create TripInvite record(s), using existing TripInvite system if present
- require recipient acceptance before they are added to the trip
- do not auto-add people to the trip
- avoid duplicate invites
- show success feedback

==================================================
FLOW B — PEOPLE NOT ON BASELODGE
==================================================

In the same modal/bottom sheet, add section:

Invite someone else

Options:

1. Share via text
2. Copy invite link

Use existing invite/share/invite-token infrastructure if present.

Generate a trip invite link/token that can be shared externally.

Expected behavior:

- if recipient already has BaseLodge, link should route them to the trip invite/accept flow
- if recipient does not have BaseLodge, link should take them through signup/onboarding first, then preserve the invite context if existing infrastructure supports it

If full external invite-token handling does not already exist, do NOT overbuild it.

Instead:
- identify what exists

- clearly note any follow-up work needed
