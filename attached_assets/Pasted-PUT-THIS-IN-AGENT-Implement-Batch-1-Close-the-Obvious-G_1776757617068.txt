## 1.1 Cancel Join Request UI

Current state:
- The backend cancel route already exists
- The friend trip detail page currently shows:
  - Request to Join
  - Request Sent
  - You’re Going
- But there is no visible way to cancel a pending join request

Required:
- Add a visible cancel action when the request is pending on `/friend-trip/<trip_id>`
- Keep the primary state clear:
  - If request is pending, show “Request Sent”
  - Also show a secondary action to cancel the request
- Keep the UI premium and clean, not noisy
- Do not expose cancel request in list views or discovery surfaces
- This action should exist only on the friend trip detail page

Behavior:
- Cancel should call the existing backend cancel route
- After cancel, the page should return to the normal “Request to Join” state
- Handle errors gracefully

## 1.4 Fix Raw 403 Page

Current state:
- Non-friend or unauthorized access to `/friend-trip/<trip_id>` can return a raw 403

Required:
- Replace raw/ugly unauthorized behavior with a friendly styled error experience
- This can be:
  - a dedicated 403 template
  - or a friendlier inline error page for this surface
- It should feel intentional and on-brand

Suggested copy direction:
- “You can’t view this trip”
- “This trip is only visible to friends or invited guests.”

Also include:
- a clear way back, such as:
  - Back to Friends
  - Back to Trips
  - Back to Home

Do not expose sensitive trip details in this state.

## 2.3 Loading States

Goal:
Add lightweight loading / pending feedback where it materially improves UX.

Important:
- This is a server-rendered app, so do not overbuild fake SPA loaders
- Focus on form/button/action feedback and any obvious page-level wait states

Add loading/pending feedback for:
- Request to Join action
- Cancel Join Request action
- Invite friend actions if applicable on trip pages
- Any obvious submit buttons on key surfaces where double-taps could occur

Requirements:
- buttons should visually enter a pending/loading state after tap/submit
- prevent accidental duplicate submits
- keep styling minimal and consistent
- do not add noisy spinners everywhere

If there are existing utility classes or JS patterns for button pending states, reuse them.

## 2.4 Activity Feed Control

Current state:
- Home activity feed can render all items with no limit/pagination
- If a user has many activity items, the screen can become noisy

Required:
- Add a reasonable default display limit to the Home activity feed
- Recommended: show the newest 8–12 items by default
- Add a simple “View more” / “Show all” behavior if needed
- Keep this implementation lightweight and mobile-friendly

Important:
- Preserve sort order (newest first)
- Do not break existing activity rendering logic
- Do not remove any activity types
- Prefer a minimal solution over building a full heavy pagination system unless the route already supports it cleanly
