#26 - Ideas Experience: Spec Alignment

Implement a spec-alignment pass for the BaseLodge Ideas experience. The previous implementation got the route structure and the broad planning/going distinction in place, but it still diverges from the finalized spec in copy, name display, rider type, ranking order, pass normalization, and availability-card framing.

This task is a correction pass, not a redesign.
Do not refactor unrelated parts of the app.
Do not change the route URLs or data model.
Use checkpoints sparingly.
Do not finish until you have fully tested the full flow end-to-end.

--------------------------------------------------
GOAL
--------------------------------------------------

Align the Ideas tab and all three detail screens to the finalized editorial spec so the UI, copy, and ranking behavior match exactly.

--------------------------------------------------
DONE LOOKS LIKE
--------------------------------------------------

1. Availability cards on the Ideas tab use:
- kicker: "AN OPENING"
- headline: "A shared opening" (not "A four-day window")
- with-line:
  - 1 overlapping friend -> "With Alex"
  - 2 overlapping friends -> "With 2 others"
  - 3 overlapping friends -> "With 3 others"
  - count overlapping friends only; exclude the current user
- date line remains date range + destination
- supporting line renders exactly:
  "On both your lists. Passes align."
  only when BOTH are true:
  1. the destination is on the current user’s wishlist
  2. passes align strongly enough to qualify
  If either condition is false, omit the supporting line entirely
- CTA remains:
  "Plan trip →"

2. Trip cards on the Ideas tab:
- planning state headline:
  "{FirstName} is considering {Destination}"
- going state headline:
  "Join {FirstName} in {Destination}"
- CTA reads:
  "Join trip →" (not "See more →")
- supporting line uses normalized pass copy:
  - exact match example: "You both have Ikon"
  - mixed example: "Passes vary"
- do NOT use "Ikon Pass" or "Epic Pass"

3. Wishlist cards on the Ideas tab:
- headline remains resort-first
- subtitle reads exactly:
  "Several of you have {Destination} saved."
  (not "Three of you have it wishlisted.")
- no date displayed
- CTA remains:
  "Suggest dates →"

4. Empty state on the Ideas tab reads:
- title: "Ideas"
- subhead: "Where your plans and your friends line up."
- body: "Nothing lined up this week."
- supporting line:
  "Set availability or add a few resorts to your list to give it something to work with."
- links:
  "Set availability →"
  "Build your list →"

5. Availability detail screen:
- title: "A shared opening"
- why line (italic): "Shared dates. Shared destination."
- participant rows show rider type + normalized pass:
  - "Skier · Ikon"
  - "Snowboarder · Epic"
- if rider type is missing, render only the normalized pass with no dangling separator
- no CTA

6. Wishlist detail screen:
- why line:
  "Several of you have {Destination} saved."
- participant rows show rider type + normalized pass
- no CTA

7. Trip detail screen:
- header ALWAYS reads:
  "Join {Name} in {Resort}"
  regardless of planning/going status
- status distinction belongs ONLY in the why line:
  - planning: "{FirstName} is considering this trip."
  - going: "{FirstName} is going."
- participant rows show rider type + normalized pass
- "You, if you join" remains as final row where appropriate
- no CTA

8. Name logic:
- Ideas tab:
  - first name only by default
  - if duplicate first names exist, use first name + last initial (e.g., Alex T.)
- Detail screens:
  - full names for other users
  - "You" for current user

9. Pass display normalization:
- strip generic suffixes like "Pass" / "pass"
- examples:
  - "Ikon Pass" -> "Ikon"
  - "Epic Pass" -> "Epic"
  - "Mountain Collective Pass" -> "Mountain Collective"
- preserve the core pass brand name
- apply normalization everywhere the short pass value appears:
  - Ideas tab supporting lines
  - detail screen participant rows

10. Ranking order is deterministic:
- shared availability > planning trips > going trips > wishlist pulls
- availability cards must outrank trip cards by base-score design, not by accident
- planning trips must outrank going trips when otherwise similar
- wishlist pulls remain lowest priority

11. All QA items pass end-to-end for demo@baselodge.app
