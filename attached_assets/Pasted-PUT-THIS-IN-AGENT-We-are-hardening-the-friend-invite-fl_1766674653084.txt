We are hardening the friend invite flow in FOUR STAGED STEPS to prevent silent failures caused by session loss.
This work should be implemented sequentially and carefully.

DO NOT skip steps.

================================
BACKGROUND (CONFIRMED)
================================

- Friend relationships are directional and require two rows.
- Friend creation logic (_connect_pending_inviter) is correct but fragile.
- The fragility comes from relying on session state to remember who invited a user.
- If the session is lost during signup, inviter information is lost and friend connections fail silently.

The fix is to make inviter linkage DURABLE and then use it safely.

================================
STEP 1 — Schema change (FOUNDATION)
================================

Add a durable inviter reference to the User model.

- Add a nullable foreign key column:
  invited_by_user_id (FK → user.id, nullable)

Requirements:
- Represents “who invited this user”
- Must persist through signup, redirects, refreshes, and session loss
- Create and run a migration
- Update the User model accordingly

This step should not change any behavior yet.

================================
STEP 2 — Capture inviter at signup (FOUNDATION)
================================

When a user signs up via an invite link:

- Resolve the inviter exactly as the code does today (invite token / session).
- When creating the new User record:
  - If an inviter is known, set:
    user.invited_by_user_id = inviter.id
- Commit this value as part of user creation.

Important:
- Continue setting session['pending_inviter_id'] as a fallback.
- Do NOT create Friend rows yet.
- This step is additive and must not break existing flows.

================================
STEP 3 — Harden _connect_pending_inviter (LOGIC)
================================

Refactor _connect_pending_inviter() to make it resilient.

Priority order when resolving inviter:
1. user.invited_by_user_id (durable, preferred)
2. session['pending_inviter_id'] (fallback)
3. Otherwise, do nothing

When an inviter is found:
- Create TWO Friend rows (bidirectional):
  - user → inviter
  - inviter → user

Make this logic IDEMPOTENT:
- Check for existing Friend rows before inserting
- Never create duplicate rows
- Safe to run multiple times

Do NOT modify the Friend feed query.

================================
STEP 4 — Execution timing (SAFETY NET)
================================

Ensure _connect_pending_inviter() is called:

- After successful signup
- After first successful login

This guarantees recovery even if the initial signup attempt failed due to session loss.
