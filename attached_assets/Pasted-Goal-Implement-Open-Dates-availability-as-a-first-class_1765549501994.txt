Goal
Implement Open Dates (availability) as a first-class Phase 1 feature. This is not a trip. It represents days a user is free to ski so friends can match availability and pass compatibility.
This prompt includes UI changes, logic, copy, editing behavior, and matching display.
1️⃣ Open Toggle – Remove “Add a trip”
Requirements
On the Open toggle (Home screen):
Remove the “+ Add a trip” button
Keep only:
“+ Add open dates” (or “Edit open dates” if dates already exist)
Rationale
The Open toggle is only for availability, not confirmed trips.
2️⃣ Open Toggle – Add descriptive copy (Option C)
At the top of the Open tab, add this copy:
Title:
Open dates
Description:
Days you’re free to ski. Add dates you’re open, see when friends are available on the same days, and whether you’re on the same pass.
This copy must appear above any empty state or cards.
3️⃣ Open Dates entry & editing (calendar-based)
Replace date inputs with a scrollable calendar
Use a tap-to-select calendar UI
Users can:
Tap a date → select
Tap again → deselect
Calendar must:
Start at today
Allow scrolling through May 2026
No start/end fields
Dates do not need to be contiguous
Editing behavior
The same calendar is used for:
Adding dates
Editing dates
No separate edit screen
Existing open dates should be pre-selected when reopening the calendar
CTA behavior
If user has no open dates:
Button label: Add open dates
If user has existing open dates:
Button label: Edit open dates
4️⃣ Data model (Phase 1 – keep simple)
Add or confirm this field on User:
open_dates = db.Column(db.JSON)  # list of YYYY-MM-DD strings
No separate table needed.
5️⃣ Open Toggle – Display logic
Section A: User summary
At the top of Open tab:
Example:
You’re open on 5 days
Dec 14, Dec 18–19, Jan 3–4
(Format dates human-readable)
Section B: Friend availability matches
Show only dates where overlap exists.
Group by date
Each card/row represents one calendar day.
Example card
Saturday, Dec 14
You + Alex are both open
Epic pass · Advanced
If passes differ:
Epic · Ikon (different passes)
Rules
A match exists if:
Same date in open_dates
Users are friends
Do not block matches due to pass mismatch
Only display compatibility, don’t enforce it
6️⃣ Pass compatibility rules (Phase 1)
For each overlapping date:
Compare current_user.pass_type vs friend.pass_type
Display:
Same pass → normal text
Different passes → label “different passes”
No filtering or scoring yet.
7️⃣ Navigation & state cleanup
Open tab must:
Not show trips
Not link to Add Trip
Trips continue to live under My Trips
Open dates are never treated as trips
8️⃣ Empty states
No open dates
You haven’t added any open dates yet.
Button:
Add open dates
No overlapping friends
You’re open on these days, but none of your friends are yet.
(Still valuable feedback — do not hide the section.)
9️⃣ Future hooks (DO NOT IMPLEMENT, just comment)
Add comments where appropriate for future enhancements:
Notifications (“2 friends are open next weekend”)
Sorting logic (same pass first)
Turning an open date into a trip
Dummy data seeding for visual QA
10️⃣ Guardrails
Do not introduce AI, scoring, or recommendations
Do not auto-create trips
Do not over-normalize the database
Keep this Phase 1 and readable
Definition of Done
Open tab shows availability, not trips
Calendar-based selection works through May 2026
Users can edit open dates easily
Overlapping friend dates are clearly displayed
Pass compatibility is visible but non-blocking
UI copy clearly explains what “Open” means
