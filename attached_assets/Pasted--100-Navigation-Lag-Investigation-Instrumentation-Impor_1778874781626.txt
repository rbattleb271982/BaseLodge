#100 — Navigation Lag Investigation & Instrumentation

Important:
This is an investigation and instrumentation task first. Do not perform broad performance refactors until timing data proves where the lag is coming from.

What & Why

Switching between screens in BaseLodge — bottom tabs, detail pages, modals, profile pages, forms, settings pages, back navigation, and native Capacitor screens — can sometimes feel delayed, jumpy, or visually laggy.

There is currently no timing instrumentation in the codebase, so the root cause is unknown.

This task adds lightweight server-side and client-side timing, identifies which screens are actually slow and why, and applies only obvious safe fixes if timing data proves them necessary.

Done looks like

- Every user-facing route logs server timing:
  route start → DB duration → render duration → total duration

- Every page records client-side navigation timing:
  click timestamp → navigation start → DOMContentLoaded → first visible content

- All instrumentation runs behind a shared debug flag

- A clear report exists identifying:
  - slowest screens
  - measured timings
  - root causes

- Small safe fixes are applied only where evidence supports them

- No UI redesign
- No transition animations
- No spinners or skeletons
- No features removed
- No console errors
- Existing navigation behavior remains intact

Out of scope

- UI redesign
- transition animation work
- skeleton loaders
- removing features
- speculative performance refactors
- fixing lag not measurable through instrumentation
