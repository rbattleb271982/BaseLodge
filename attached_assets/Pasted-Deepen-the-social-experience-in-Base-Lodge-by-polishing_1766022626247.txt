Deepen the social experience in Base Lodge by polishing Friend Profiles, surfacing Trip Overlaps, and adding lightweight Pass Compatibility indicators. This phase focuses on people and shared moments — no new navigation paradigms, no heavy utilities.

Scope
Friends list → Friend profile detail
Home + Friend profiles → Trip overlaps
Trips + Friend profiles → Pass compatibility indicators

No new data models. No routing changes unless required for friend profile navigation. No redesign of global navigation.

––––––––––––––––––
	1.	Friend Profile Page (Phase 1 Priority #1)
––––––––––––––––––

When a user taps a friend from the Friends list, show a clean, intentional friend detail view.

Structure (top to bottom):

A) Header (Card)
– Friend name (primary)
– Rider type · Skill level · Passes (secondary)
– Home mountain (if available)

Rules:
– This is a “person moment” → card is appropriate
– Keep typography calm and breathable
– Do not overload with stats

B) Equipment (Flat list section)
– Display equipment using the same category-first rules as the user profile
– Skis/Board + Boots
– Collapsed by default, expandable inline
– Omit missing categories

C) Upcoming Trips (Carded list)
– Show friend’s upcoming trips using the existing trip card pattern
– No emojis
– Two-line hierarchy
– If no upcoming trips, show a friendly empty state

D) Shared Context (Optional if data exists)
– Shared mountains visited
– Shared trips (if applicable)
– Keep this lightweight, list-style

Do NOT:
– Add messaging
– Add invite flows
– Add new CTAs

This screen should answer: “Who is this person and how do they ski?”

––––––––––––––––––
2) Trip Overlap Cards (Phase 1 Priority #2)
––––––––––––––––––

Surface overlapping trips with friends in a human, obvious way.

Definition:
A trip overlap exists when two users are at the same resort with overlapping dates.

Where to show overlaps:

A) Home
– If overlaps exist, show a small card or callout near the top of the Trips section

B) Friend Profile
– Show overlaps between current user and that friend

C) Trip Detail (if applicable)
– Show which friends overlap on this trip

Copy style (examples):
– “You’ll both be at Breckenridge · Dec 21–23”
– “You and Alex overlap at Park City · Jan 4–6”

Rules:
– Use plain, human language
– One overlap per line or card
– No icons or emojis
– Do not overwhelm the screen

Overlaps should feel like insight, not alerts.

––––––––––––––––––
3) Pass Compatibility Indicators (Phase 1 Priority #3)
––––––––––––––––––

Add lightweight, contextual indicators to show where friends can ski together based on passes.

Rules:
– Passive indicators only (no badges everywhere)
– Show only where relevant

Where to apply:

A) Friend Profile
– If a friend can ski at the same resorts as the user’s upcoming trips, show subtle copy like:
“Can ski your upcoming trips”

B) Trip Context
– If viewing a trip, show:
“3 friends can ski here with their pass”

Do NOT:
– Build a full pass matrix
– Add icons per pass
– Add new filters

This should feel helpful, not technical.

––––––––––––––––––
Do NOT Do (Phase Guardrails)
––––––––––––––––––

– Do not add Resort Browser
– Do not redesign calendar picker
– Do not expand group trip flows
– Do not add new navigation tabs
– Do not add animations

––––––––––––––––––
Acceptance Criteria
––––––––––––––––––

– Friend profiles feel rich but calm
– Tapping a friend feels rewarding
– Trip overlaps are obvious and human-readable
– Pass compatibility is clear without visual clutter
– App feels more social without feeling more complex
