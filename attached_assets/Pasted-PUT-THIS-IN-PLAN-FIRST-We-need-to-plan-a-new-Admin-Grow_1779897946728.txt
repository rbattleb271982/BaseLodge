Goal:
Create a clear admin information architecture where each page has a distinct purpose and we avoid unnecessary duplicate information.

Important philosophy:
The main Dashboard should be minimal and high-level.
The most important executive summary metrics can live on Dashboard, but deeper analysis should live on the specialized pages.

It is okay if a high-level metric appears on Dashboard and also appears in more detail elsewhere.
Example:
- Dashboard can show “Push Opt-In %”
- Messaging owns the detailed Push Health section

But avoid duplicating full sections, tables, charts, and analysis across multiple admin pages.

Proposed admin page roles:

1. Dashboard
Purpose:
Executive snapshot of BaseLodge health.
Should be minimal, high-level, and cross-functional.
Should link users to deeper pages instead of duplicating those pages.

2. Growth
Purpose:
Is BaseLodge growing into a healthier, denser social network?
Focus on:
- acquisition
- activation
- retention
- invite conversion
- onboarding completion
- network effects
- friend graph density
- social conversion

Do NOT turn Growth into:
- Messaging diagnostics
- Trips operations
- Mountain traffic dashboard
- CRM/campaign reporting
- predictive analytics
- generic vanity metric wall

3. Messaging
Purpose:
Communication system health.
Owns:
- push health
- email preferences
- delivery health
- suppression/retries
- lifecycle communication events
- raw message log link

4. Trips
Purpose:
Trip operations and trip planning behavior.
Owns:
- trips created
- upcoming/completed trips
- party size
- solo vs group
- invite behavior tied to trips
- trip destination patterns

5. Mountains
Purpose:
Destination and resort engagement.
Owns:
- mountain page views
- destination popularity
- wishlist/overlap destination intelligence
- resort-level engagement

6. User Insights
Purpose:
User base composition and profile completeness.
Owns:
- user segments
- onboarding/profile completeness
- rider type/pass/equipment/preference breakdowns if present
