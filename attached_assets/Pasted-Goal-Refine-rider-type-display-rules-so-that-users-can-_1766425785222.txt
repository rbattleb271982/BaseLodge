Goal:
Refine rider type display rules so that users can select multiple rider types (Primary + Secondary), but the app displays only the Primary rider type in compact surfaces, and shows both Primary and Secondary only in full profile/detail views. Do not introduce any “Both” option.

Decisions (locked):
	•	There is no “Both” rider type
	•	Rider combinations are always explicit (e.g., Skier + Snowboarder)
	•	Primary rider type represents the user’s main identity
	•	Secondary rider types add nuance but are not always shown
	•	Display behavior depends on context

Authoritative Rider Type List (exact values only):
	•	Skier
	•	Snowboarder
	•	Telemark
	•	Cross-Country
	•	Adaptive

Data Model (unchanged):
	•	primary_rider_type (String, required)
	•	secondary_rider_types (JSON, optional, max 2)
	•	Legacy rider_type remains deprecated but intact

Display Rules:
	1.	Compact / summary surfaces (PRIMARY ONLY)
Show only primary_rider_type in these locations:

	•	Home profile card
	•	Friend tiles
	•	Friend list rows
	•	Invite cards
	•	Invite landing summaries
	•	Any other space-constrained identity row

Example outputs:
	•	“Skier”
	•	“Snowboarder”

Ignore secondary_rider_types entirely in these contexts.
	2.	Profile / detail surfaces (PRIMARY + SECONDARY)
Show combined rider types only on:

	•	Full profile page
	•	Profile edit / settings view
	•	Any explicit “About” or identity detail section

Formatting:
	•	If only primary exists: “Skier”
	•	If secondary exists: “Skier · Snowboarder”
	•	Use middle dot separator
	•	Preserve order: Primary first, then Secondary values in stored order

	3.	Implementation guidance:

	•	Do not create a single global display helper unless it supports a context flag (e.g., compact vs full)
	•	Ensure no screen ever renders JSON or array syntax
	•	Ensure seeded users without secondary data still render correctly

	4.	Explicit constraints:

	•	Do not add a “Both” option
	•	Do not auto-generate combined labels
	•	Do not change filtering or matching logic yet
	•	Do not expose “Primary” or “Secondary” labels publicly

Acceptance Criteria:
	•	Compact views show exactly one rider type (Primary)
	•	Profile views show both if applicable
	•	No existing screens regress visually
	•	No ambiguity in rider identity anywhere in the app
