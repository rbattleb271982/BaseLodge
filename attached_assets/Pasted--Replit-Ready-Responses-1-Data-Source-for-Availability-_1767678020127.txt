1. Data Source for Availability
Use the existing Open Availability data.
There is already an “Open Availability” concept in the app (this is what Planning is based on).
Do NOT derive availability from SkiTrip records.
Planning should be based on explicit availability, not inferred intent.
If there is an existing OpenAvailability / Availability model or table, use that directly.
If availability is stored in a user-associated table with start/end dates, that is the source of truth.
⚠️ Do not treat “planning trips” or draft trips as availability.
2. Window Grouping Logic
Use Option (a):
Show the intersection with the current user’s availability as a single card.
Details:
Planning cards represent overlap windows between the current user AND friends
If:
User is free Jan 18–22
Friend A: Jan 18–21
Friend B: Jan 19–22
Then the overlap windows should be:
Jan 18–21 (Friend A)
Jan 19–22 (Friend B)
If multiple friends share the same overlap window, group them into one card with a count.
Do not:
Show raw friend windows
Merge windows beyond true overlaps
Create “unioned” ranges that the user is not fully free for
Rule of thumb:
A Planning card only exists where BOTH the user and at least one friend are free.
3. Current User’s Availability Required?
✅ Yes — the current user must have availability set.
If the user has no availability:
Show an empty state:
“Add your availability to see when friends are free.”
Do not show all friend availability in this case.
Planning is about alignment, not browsing others’ schedules.
4. Navigation Placement

This is a first-class surface, not a sub-section

Final bottom nav:

Use the existing format_trip_dates() helper.
