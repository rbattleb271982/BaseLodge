You are working on the BaseLodge Flask web app. Implement the following feature end-to-end, testing all flows before finishing. Use checkpoints sparingly.

Goal
Add explicit trip duration and equipment status so users can easily coordinate logistics, discover who is going up for a day trip, filter trips by compatibility, prevent duplicate overlapping trips, and support future notifications. This is a social coordination feature, not just metadata.

Scope
This work applies to the SkiTrip model and the user Profile only. Do not modify or extend GroupTrip in this iteration.

––––––––––––––––
TRIP DURATION
––––––––––––––––

Data Model
Update the SkiTrip model to include a new required field called trip_duration.

Allowed values (exact):
DAY_TRIP
ONE_NIGHT
TWO_NIGHTS
THREE_PLUS_NIGHTS

Display labels:
Day Trip
1 Night
2 Nights
3+ Nights

Backfill Existing Trips
There are existing trips in the database. Backfill trip_duration automatically using current start_date and end_date.

Rules:
Same start and end date → DAY_TRIP
1 night → ONE_NIGHT
2 nights → TWO_NIGHTS
3 or more nights → THREE_PLUS_NIGHTS

Do not make this field nullable.

Create Trip Behavior
Users must explicitly select a trip duration.
Do not infer duration from dates on creation.
Block save if duration is missing.

Edit Trip Behavior
If trip dates change, auto-update trip_duration using the same rules as backfill.
Do not show a confirmation modal.
Update silently.

Active Trip Definition
A trip is active if end_date >= today.
Upcoming and in-progress trips are active.
Past trips (end_date < today) are inactive and must be excluded from overlap checks, social counts, and filters.
Treat dates as date-only. Ignore timezones.

Overlapping Trip Validation
Allow only one active SkiTrip per user per resort per overlapping date range.

Rules:
Overlap checks apply only to active trips.
Enforce per resort.
Multiple trips to different resorts in the same weekend are allowed.

If a user attempts to create or edit a trip that overlaps with an existing active trip at the same resort, block the action and show this alert:

“You already have a trip to {Resort Name} that overlaps these dates. Want to update that trip instead?”

Buttons:
Edit existing trip → redirect in the same tab to the edit page of the overlapping trip
Cancel → remain on the current form

Do not allow duplicate active trips.

Trip Card Display
Wherever SkiTrips appear (home, trip lists, friend cards), show duration inline and clearly.

Examples:
Sat, Jan 13 · Day Trip
Jan 13–14 · 1 Night

Duration must be visible without clicking and more prominent than dates in social contexts.

Home Screen Day-Trip Signal
Add a subtle home screen card that answers “Who’s doing a day trip?”

Weekend definition:
Friday through Sunday.

Weekend logic:
If today is Monday–Thursday, use the upcoming Friday–Sunday window.
If today is Friday–Sunday, use the current Friday–Sunday window.

Rules:
Destination name must always be shown.
Example copy: “3 friends are day-tripping Breckenridge this weekend”.
Only show the card if the count is greater than zero.
Place the card above the main tabs with other home summary cards.
Do not include a CTA button.

EQUIPMENT STATUS
––––––––––––––––

Profile-Level Equipment Status
Add a required profile field called equipment_status.

Allowed values (exact):
HAVE_OWN_EQUIPMENT
NEEDS_RENTALS

Display labels:
Have own equipment
Needs rentals

This field represents whether the user owns equipment in general and is used as the default for trips.

Interaction With Existing Equipment Brand Fields
If equipment_status is HAVE_OWN_EQUIPMENT:
Show existing equipment brand fields normally.

If equipment_status is NEEDS_RENTALS:
Completely hide all equipment brand fields.
Do not show disabled inputs, placeholders, or helper text.
Do not require users to interact with brand fields in any way.

Do not delete or modify existing brand data in the database. Simply hide or show fields based on equipment_status.

Trip-Level Equipment Override
On SkiTrip create/edit, add an optional field called “Equipment for this trip”.

Options:
Use my default
Have own equipment
Needs rentals

Default selection is “Use my default”.

Behavior:
If set to “Use my default”, inherit from profile equipment_status.
If overridden, use the trip-specific value for that trip only.
Trip-level equipment status should never force users to select brands.

Trip Card Display (Equipment)
On trip cards and lists, show equipment status inline with duration.

Examples:
Day Trip · Have own equipment
1 Night · Needs rentals

Never display equipment brand names on trip cards or social surfaces.
